### AUDIT-01 - "Every operation is idempotent" asserted as absolute fact, contradicted on the same page
* **Source Document:** `docs/foundations/communication-flow.md`
* **Problematic Text:**
  > "Duplicates are acceptable because every operation is idempotent on its `requestId`."
* **First-Principles Violation:** Unsubstantiated Assumption. This states as universal fact that every operation is idempotent on its requestId, but nothing substantiates it and the same page (line 59) says reboot "is asynchronous and returns before the device actually reboots" and set_os only acknowledges immediately — retrying these with the same requestId is not effect-idempotent. The page also later reframes idempotency as a discipline the application must build ("Treat commands as idempotent" / "Build retry around requestId reuse"), so the absolute claim is unproven. CONFIRMED genuine: idempotency varies by command — read-only get_* are idempotent, but set_*/control_operation/firmware are state-changing and must be retried with a NEW requestId.
* **Mapped Meta References:**
  - `_meta/knowledge-base/product/reference/handheld-rfid-iotc-master-docset/zebra-handheld-rfid-iotc-content.md` (Directly resolves the issue. Its idempotency guidance (around line 1062) establishes that idempotency varies by command: read-only commands such as get_status and get_version are idempotent, while state-changing commands such as set_operating_mode and config_endpoint are not idempotent in the strict sense (replaying them means the last one wins), and a firmware update must not be retried blindly. Lines 3434-3435 give the correct retry contract: idempotent get_* retry with the SAME requestId; state-changing set_*/control_operation retry with a NEW requestId. This flatly refutes the page's universal "every operation is idempotent on its requestId" and supplies the correct per-command rule.)
  - `_meta/knowledge-base/product/reference/handheld-rfid-iotc-api-schema-and-docs-technical-writer/operation_descriptions/set_os.md` (SoT proof that at least one operation is NOT idempotent on its requestId. Section 6 (line 77): "A firmware update cannot be started while another update is already in progress. Sending set_os during an active update returns error code 4. Wait for the current update to complete before retrying." Retrying set_os with the same requestId yields a different result (error code 4) rather than the same outcome, so the page's absolute idempotency claim is provably wrong for firmware updates.)
  - `_meta/knowledge-base/product/reference/handheld-rfid-iotc-api-schema-and-docs-technical-writer/operation_descriptions/reboot.md` (SoT confirming reboot is a stateful warm reset, not a requestId-deduplicated no-op. It describes a "warm reset of the device" with automatic server reconnection and notes config side effects (radio operation configurations from control endpoint operations are lost on reboot). There is no reader-side requestId dedup; a repeated reboot re-fires the reset, substantiating the same-page observation (line 59) that reboot returns before the device actually reboots and is therefore not effect-idempotent.)
* **Status:** Verified

This page itself calls the scanner 'a separate subsystem from the RFID radio,' yet asserts these radio/inventory codes apply to SCANNER START/STOP with no SoT basis (the page's SME note explicitly excludes the command mechanics from its caveat, calling them 'schema-accurate').   - `_meta/knowledge-base/product/reference/handheld-rfid-iotc-api-schema-and-docs-technical-writer/error_codes.json` (The authoritative error-code registry. Code 11 = 'Inventory in progress' (iot_status_code IOT_ERROR_INVENTORY_IN_PROGRESS; cause 'An RFID inventory operation is currently running') and code 12 = 'No Radio Operation in Progress' (iot_status_code IOT_ERROR_NO_RADIO_OP_IN_PROGRESS; cause 'A stop command was sent but no radio operation is currently active'; recommended_action 'the radio is already idle').
