## Overview

`dataEVT` is the reader's **tag-data** event — the device publishes it **autonomously** while an inventory is running, carrying the RFID tag reads (and any barcode reads) it produces. It is **device-emitted** (no command is in the payload path), but it only flows once you **start an inventory** with [`control_operation`](/reference/ctrl/control-operation) `START`, using whatever the [`set_operating_mode`](/reference/ctrl/set-operating-mode) profile configured. It is read-only tag output.

Each event is a small envelope — `type` (the active profile), `timestamp`, and a `data` object holding a **`tagData[]`** array (and/or a `barcodeData[]` array). Reads are **batched** (up to ~40 tags per event), and **which fields each tag carries is gated by the operating mode's `tagMetaDataToEnable` flags** — only `EPCid` and `eventNum` are effectively always present.

:::note[Flows on the DATA endpoint, not the management/control planes]
`dataEVT` is published on the **DATA endpoint's** data topic (e.g. `zebra/DATA/clients/data1event/{serial}` for `DATA1`), not the `event` topic ([`alerts`](/reference/events/alerts-event) / [`heartbeatEVT`](/reference/events/heartbeat-event)) or the `rfid` topic ([`mqttConnEVT`](/reference/events/mqtt-connection-event)). Receiving it requires an **active DATA endpoint** ([`config_endpoint`](/reference/mgmt/config-endpoint), `epType: DATA1`/`DATA2`). Two parallel channels — `DATA1` and `DATA2` — can each carry `dataEVT`.
:::

## Topic Mapping

`dataEVT` is device-originated and has no `cmnd`/`resp` topics. It is published on the DATA endpoint's data topic.

| Direction | Topic | Note |
| --- | --- | --- |
| Device-emitted (DATA1) | `zebra/DATA/clients/data1event/{serial}` | The first data channel's tag stream (verified on-device). |
| Device-emitted (DATA2) | `zebra/DATA/clients/data2event/{serial}` | The second data channel, when a `DATA2` endpoint is configured. |

The full pattern is `<tenantId>/<topic>/<deviceSerialNumber>`; the exact middle segment is the DATA endpoint's configured `publishTopic`. See [Topic hierarchy](/foundations/mqtt/topic-hierarchy) and the [Topic quick reference](/reference/appendices/topic-quick-reference).

## How reads are shaped

Two operating-mode settings determine what each `dataEVT` carries:

- **`tagMetaDataToEnable`** (in [`set_operating_mode`](/reference/ctrl/set-operating-mode)) gates the per-read fields: `RSSI` → `peakRssi`, `PHASE` → `phase`, `CHANNEL` → `channel`, `SEENCOUNT` → `seenCount`, `EPC`/`TID`/`USER`, `PC`, `MAC`, `HOSTNAME`. `EPCid` is always present. (`XPC`/`CRC` enable flags may not be honored — see Notes.)
- **`reportFilter` duration** controls aggregation: at **`0`**, every individual read is reported (`channel` and `phase` are included, and `seenCount` is `1`); at **> 0**, reads are aggregated and reported at intervals, `seenCount` counts the reads since the last report, and `peakRssi` is the peak since the last report.

## Notes & Caveats

:::note
**Field presence is gated; only `EPCid` + `eventNum` are effectively always there.** Beyond `EPCid` (always present) and the required `eventNum`, every `tagData` field appears only when its `tagMetaDataToEnable` flag is set in [`set_operating_mode`](/reference/ctrl/set-operating-mode). A default-mode read typically carries just `EPCid`, `eventNum`, `peakRssi`, and `seenCount`. Code defensively — treat all other fields as optional.
:::

:::warning[`channel` is a raw index, not MHz]
Despite the schema describing `channel` as a value "in MHz", the device emits a **raw channel value/index** (e.g. `9216`), **not** a ~900 MHz frequency. Do not interpret `channel` as megahertz; treat it as an opaque channel index. `channel` (and `phase`) are reported only when `reportFilter` duration = 0.
:::

:::note
**`XPC` and `CRC` enable flags may be ignored.** On current firmware, requesting `XPC` or `CRC` in `tagMetaDataToEnable` may read back as enabled-but-not-honored — they were never emitted in testing. `XPC` is reported only if physically present on the tag. Don't rely on `XPC`/`CRC` being present even when enabled.
:::

:::note
**`accessResults` appears only for configured access operations.** `accessResults` is populated only when you configure `accessOperations` (READ/WRITE/LOCK/KILL) in [`set_operating_mode`](/reference/ctrl/set-operating-mode); each string is one op's outcome (e.g. `READ-EPC-Success`, or an error/`SUCCESS` for writes). A bank read enabled purely via `tagMetaDataToEnable` surfaces as the `EPC`/`TID`/`USER` metadata fields, not as `accessResults`.
:::

:::warning[`RESERVED`-bank reads can expose tag passwords]
The `RESERVED` memory bank holds a tag's **access and kill passwords**. If you configure a `READ` of the RESERVED bank, those passwords appear in `dataEVT` as the `RESERVED` hex field — treat that data as sensitive. (`EPCid` and the other bank contents are physical-tag data, not device or broker credentials.)
:::

:::note
**`STOP` is not instantaneous for buffered reads.** [`control_operation`](/reference/ctrl/control-operation) `STOP` halts the radio, but a buffered in-flight `dataEVT` batch may still be delivered a few seconds after `STOP` (≈3–5 s observed on RFD40). Don't treat a `dataEVT` arriving shortly after `STOP` as an error, and don't stop listening the instant you send `STOP`.
:::

:::note
**Batching, `type`, and `FAST_READ`.** Reads are batched — up to ~40 tags per `dataEVT`. `type` echoes the active operating-mode profile (e.g. `BALANCED_PERFORMANCE`); the reader emits the same profile on every event of a run. Note the `FAST_READ` profile does **not** emit `dataEVT` on current firmware (see [`set_operating_mode`](/reference/ctrl/set-operating-mode)), so `type` will not appear as `FAST_READ` on a data event. `dataEVT` carries **no `response`/`code`** — the command response-code table does not apply, and `error_codes.json` lists no codes for events.
:::

:::tip
Start the stream with [`control_operation`](/reference/ctrl/control-operation) `START` and stop it with `STOP`. Because `dataEVT` reflects the operating mode, **any armed destructive `accessOperations` (WRITE/LOCK/KILL) execute on the tags read during the run** — confirm the mode is read-only before starting (see [`set_operating_mode` § Notes](/reference/ctrl/set-operating-mode#notes--caveats)). For the concept-level view, see [Where tag reads come from](/rfid/dataevt-schema), [Two data channels](/rfid/tag-data/dual-channels), and [Interpreting tag reads](/rfid/tag-data/interpret).
:::
