## Overview

`alerts` is a **device-emitted** event that delivers structured, categorized alert information — the reader publishes it **autonomously** when a monitored condition changes state or crosses a threshold; **no command triggers it**. Each event is a fixed **envelope** (`type`, `timestamp`, `state`, `id`, `priority`) plus a category-specific **`alertDetails`** body selected by `id`. It is read-only telemetry and carries no credentials.

The same envelope carries two message classes, distinguished by `type`:

- **`ALERT`** — an autonomous monitoring alert (battery, power, network).
- **`NOTIFICATION`** — a command side-effect: `FILE_DOWNLOAD` (see [`install_certificate`](/reference/mgmt/install-certificate)) and the `FIRMWARE_UPDATE` start (see [`set_os`](/reference/mgmt/set-os)).

This page documents the full envelope and **all six `alertDetails` branches**. Which alerts publish is governed by [`config_events`](/reference/mgmt/config-events) toggles (`battery`, `power`, `network`, `firmwareUpdate`, `fileDownload`).

:::note[Supported vs not-yet-emitted]
Five `id` categories are emitted on current V1.1 firmware: **`BATTERY`**, **`POWER`**, **`NETWORK_EVENT`**, **`FIRMWARE_UPDATE`**, and **`FILE_DOWNLOAD`** (all live-verified). **`TEMPERATURE`** is declared in the `id` enum but **not emitted** on current firmware (the schema itself notes antenna, temperature, exceptions, CPU-usage, GPI, and user-app events are not supported). See [The IOTC event model](/observability/events/model).
:::

## Details

| Property | Value |
| --- | --- |
| **Event Type** | Alert / notification — structured, categorized |
| **Direction** | Device → application (device-emitted; no request/response) |
| **Applies To** | RFD40 series, RFD90 series (BATTERY / POWER / NETWORK_EVENT / FIRMWARE_UPDATE / FILE_DOWNLOAD verified on RFD40) |
| **Trigger** | Autonomous, on a monitored state change or threshold crossing |
| **Enabled by** | [`config_events`](/reference/mgmt/config-events) `eventConfiguration` toggles (per category) |
| **Related Events** | [`heartbeatEVT`](/reference/events/heartbeat-event), [`dataEVT`](/reference/data/tag-data-event), [`mqttConnEVT`](/reference/events/mqtt-connection-event) |
| **Credentials** | None — read-only telemetry, nothing to mask |

## Topic Mapping

`alerts` is device-originated and flows on the **management event topic** — it has no `cmnd`/`resp` topics because no command produces it.

| Direction | Topic | Note |
| --- | --- | --- |
| Device-emitted event (subscribe by app) | `zebra/MDM/clients/event/{serial}` (or `zebra/MGMT/clients/event/{serial}`) | The same event topic that carries `heartbeatEVT` and `NOTIFICATION`-type events. |

The full pattern is `<tenantId>/<topic>/<deviceSerialNumber>`. See [Topic hierarchy](/foundations/mqtt/topic-hierarchy) and the [Topic quick reference](/reference/appendices/topic-quick-reference).

## Event payload

The envelope (`type`, `timestamp`, `state`, `id`, `priority`) is always present and **required**; `alertDetails` is optional and carries exactly one branch matching `id`. Use the **Schema** tab for the full field reference across all six branches and the **Example** tab for a battery alert.

:::note
`state` tells you whether a condition is active or resolved: **`SET`** = active/asserted, **`CLEAR`** = resolved, **`ONESHOT`** = a one-time event with no persistent state (it fires once and has no matching `CLEAR`). Live battery/power/network alerts are `ONESHOT`; a firmware update in progress is `SET`.
:::

## alertDetails branches

`alertDetails` is an `anyOf` over `alertDetails.yaml`; the populated key is determined by `id`:

| `id` | `alertDetails` key | Body fields | Emitted on V1.1 |
| --- | --- | --- | --- |
| `BATTERY` | `batteryAlert` | `status`, `stateOfHealth`, `chargePercentage` | ✅ |
| `POWER` | `powerEvent` | `powerSource`, `powerMode`, (`lldp`) | ✅ |
| `NETWORK_EVENT` | `networkInfo` | `networkInterface.{ethStatus[], wifiStatus[]}`, `btRemoteDevice` | ✅ |
| `FIRMWARE_UPDATE` | `fwUpdateStatus` | `updateStatus`, `overallProgress`, `stage` | ✅ |
| `FILE_DOWNLOAD` | `downloadInfo` | `fileType`, `fileName`, `status` | ✅ |
| `TEMPERATURE` | `temperatureInfo` | `nge`, `pa`, `ambient` | ⚠️ not emitted (see Notes) |

## Notes & Caveats

:::note
**`TEMPERATURE` is declared but not emitted.** The `id` enum includes `TEMPERATURE` (routing to `temperatureInfo` → `nge`/`pa`/`ambient`), but current V1.1 firmware does not emit temperature alerts — the schema description explicitly lists temperature (with antenna, exceptions, CPU-usage, GPI, and user-app) among the unsupported categories. The `temperatureInfo` field descriptions in the schema are placeholders; the intended meanings are `nge` = NGE/radio temperature, `pa` = power-amplifier temperature, `ambient` = ambient temperature.
:::

:::warning[Network interface field name: `interface` vs `interfaceName`]
On the wire (and in the canonical `alerts.json` examples), each `ethStatus`/`wifiStatus` item's interface name is the key **`interface`** (e.g. `"interface": "eth0"`). The leaf schema `networkEvent.yaml`, however, declares the property as **`interfaceName`** and lists it in the item `required` array. This is an internal schema inconsistency — **key on `interface`**, which is what the device emits.
:::

:::note
**`type` distinguishes alerts from notifications.** `type: ALERT` is an autonomous monitoring alert; `type: NOTIFICATION` is a command side-effect — `FILE_DOWNLOAD` (during [`install_certificate`](/reference/mgmt/install-certificate) / firmware download) and the `FIRMWARE_UPDATE` *start* (during [`set_os`](/reference/mgmt/set-os)). `EVENT` is in the enum but was not observed. All three share this one envelope.
:::

:::note
**`FILE_DOWNLOAD` `fileName` is the stored basename.** `downloadInfo.fileName` is the name the reader stores internally — the URL basename **stem** (e.g. a source `…/mqtt_ca_cert.pem` yields `fileName: "mqtt_ca_cert"`), not your request name. Don't expect a 1:1 match to a [`get_installed_certificate`](/reference/mgmt/get-installed-certificate) entry name without the source URLs.
:::

:::note
**Validate against the canonical `alerts.json`.** Three alert schemas exist in the source tree: **`events/alerts.json`** (this one — the long-form envelope the device emits; canonical), `events/alert_short.json` (the **SOTI** flat projection with granular ids like `BATTERY_LOW_SET` — out of scope here), and `refrence/events/alertEvent.yaml` (a legacy/alternate shape). Only `alerts.json` matches what the reader publishes.
:::

:::note
**`alertDetails` is optional; envelope is required.** The envelope `required` set is `[type, timestamp, state, id, priority]`; `alertDetails` is not required. Validate the envelope strictly; the branch bodies now match firmware (the earlier over-strict `required` arrays and the missing `powerEvent` branch have been resolved), with the one `interface` naming caveat above.
:::

:::note
**No response envelope.** `alerts` is a device-emitted event, so it carries **no `response`/`code`** — the command response-code table does not apply, and `error_codes.json` lists no codes for events.
:::

:::tip
`alerts` is **read-only telemetry** with no credentials. Remember emission is **gated by [`config_events`](/reference/mgmt/config-events)** — a disabled category never publishes, so the *absence* of an alert is not proof of a healthy state. For the concept-level view, see [When the reader needs to interrupt you](/observability/alerts) and [Choose what the reader tells you](/observability/configure-events).
:::
