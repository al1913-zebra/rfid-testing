> 📕 **REFERENCE** · **Audience:** API consumer · **Use:** track each endpoint's connect / disconnect transitions

## Overview

`mqttConnEVT` is a **device-emitted** event that reports an MQTT endpoint's **connection-state transition** — `CONNECTED` or `DISCONNECTED` — along with device identity (`deviceModel`, `deviceSerialNo`) and protocol-version metadata (`apiVersion`, `mqttVersion`). The reader publishes it **autonomously** on every connection-state change; **no command triggers it and no configuration enables it**. It is read-only telemetry and carries no credentials.

It behaves like standard MQTT connection lifecycle messages: a **`CONNECTED`** event is a *birth* message published when an endpoint (re)connects, and a **`DISCONNECTED`** event is a *Last-Will* message the broker publishes when the device session drops. **Each active endpoint emits its own** `mqttConnEVT`.

:::note[Published on the per-endpoint `rfid` topic — not the event topic]
`mqttConnEVT` is published on each endpoint's **`{EP}/clients/rfid`** topic (e.g. `zebra/MDM/clients/rfid/{serial}`, `zebra/CTRL/clients/rfid/{serial}`) — **not** the `{EP}/clients/event` topic that carries [`alerts`](/reference/events/alerts-event) and [`heartbeatEVT`](/reference/events/heartbeat-event), and not the DATA topic that carries [`dataEVT`](/reference/data/tag-data-event). Subscribe to the `rfid` topic of each endpoint you want connection events for.
:::

## Details

| Property | Value |
| --- | --- |
| **Event Type** | Connection-state transition (birth / Last-Will) |
| **Direction** | Device → application (device-emitted; no request/response) |
| **Applies To** | RFD40 series, RFD90 series (verified on RFD40) |
| **Trigger** | Autonomous, on every MQTT endpoint connect / disconnect — no command, no config |
| **Topic** | Per-endpoint `zebra/{EP}/clients/rfid/{serial}` (MDM, CTRL, …), `retain=false` |
| **Related Events** | [`alerts`](/reference/events/alerts-event), [`heartbeatEVT`](/reference/events/heartbeat-event), [`dataEVT`](/reference/data/tag-data-event) |
| **Credentials** | None — read-only telemetry, nothing to mask |

## Topic Mapping

`mqttConnEVT` is device-originated and has no `cmnd`/`resp` topics. It is published on each active endpoint's own `rfid` topic.

| Direction | Topic | Note |
| --- | --- | --- |
| Device-emitted (per endpoint) | `zebra/MDM/clients/rfid/{serial}` | The MDM/management endpoint's connection events. |
| Device-emitted (per endpoint) | `zebra/CTRL/clients/rfid/{serial}` | The CTRL endpoint's connection events (each active endpoint emits its own). |

The full pattern is `<tenantId>/<topic>/<deviceSerialNumber>`. On a reboot-triggered reconnect, each active endpoint (MDM, CTRL, …) publishes its own `CONNECTED` event on its own `rfid` topic. See [Topic hierarchy](/foundations/mqtt/topic-hierarchy), the [Connection lifecycle](/foundations/mqtt/connection-lifecycle), and the [Topic quick reference](/reference/appendices/topic-quick-reference).

## Connection semantics

| `connectionState` | MQTT role | When | `timestamp` |
| --- | --- | --- | --- |
| `CONNECTED` | Birth message | Published by the endpoint when it (re)connects to the broker. | Present (ISO-8601). |
| `DISCONNECTED` | Last-Will (LWT) | Published by the broker when the device session drops (detected via keepalive). | Usually **omitted** — the device is gone, so it cannot stamp it. |

Because `DISCONNECTED` is a broker-published Last-Will, it can arrive some seconds after the actual drop (broker keepalive detection latency — ~26 s observed on RFD40 after a reboot), and it carries no `timestamp`. `CONNECTED` events follow on reconnect, each with its own ISO-8601 `timestamp`.

## Notes & Caveats

:::note
**`apiVersion` is a bare number, not the `V…` string.** `mqttConnEVT` reports `apiVersion` as a JSON **number** (e.g. `1.21`). This differs from command responses, which report `apiVersion` as a **string** with a leading `V` (e.g. [`get_version`](/reference/mgmt/get-version) returns `V1.21`). Same version family, different representation — don't compare them as equal strings.
:::

:::warning[`mqttVersion` is a float artifact of 3.1]
The device serializes `mqttVersion` as the JSON number **`3.0999999`**, a floating-point representation artifact of **`3.1`** (MQTT 3.1). Treat any `mqttVersion` value ≈ `3.1` as protocol version 3.1; don't test it for exact equality to `3.1`. (This is a known firmware serialization quirk.)
:::

:::note
**`timestamp` is optional and format `date-time`.** When present it is a full ISO-8601 instant (e.g. `2026-06-15T13:09:33.810Z`). It is typically **omitted on `DISCONNECTED`** (a broker Last-Will published after the device is gone), which the schema permits because there is no `required` array.
:::

:::note
**Per-endpoint emission, `retain=false`.** Each active endpoint publishes its own `mqttConnEVT` on its own `{EP}/clients/rfid` topic, so a single reconnect yields one event per endpoint (e.g. MDM and CTRL each publish a `CONNECTED`). All are published with `retain=false`, so a late subscriber won't see the last state replayed — subscribe before the transition you want to observe.
:::

:::note
**No response envelope; automatic.** `mqttConnEVT` is a device-emitted event, so it carries **no `response`/`code`** — the command response-code table does not apply, and `error_codes.json` lists no codes for events. It needs no `config_events` toggle; the reader always emits it on a connection-state change. (The schema file's `title` is `epConnection`, a cosmetic mismatch with the event name — the on-the-wire payload is unaffected.)
:::

:::tip
`mqttConnEVT` is **read-only telemetry** with no credentials (`deviceSerialNo` is an identifier; `apiVersion`/`mqttVersion` are version values). Use the `rfid`-topic connection events for a reliable presence/liveness signal — the `DISCONNECTED` Last-Will tells you a device dropped even when the device itself can't. For the concept-level view, see [Knowing when you're connected](/observability/mqtt-connection) and [Connection lifecycle](/foundations/mqtt/connection-lifecycle).
:::
