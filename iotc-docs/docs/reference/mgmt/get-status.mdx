## Overview

`get_status` retrieves a point-in-time health and status snapshot of the reader. It is a read-only query on the device-management (MGMT) endpoint: the app publishes the command and the device publishes a correlated reply containing power, radio, clock, terminal-link, and battery information under `deviceStatus`.

Because it changes nothing on the device, `get_status` is safe to call at any time. It is the on-demand counterpart to the asynchronous `heartbeatEVT` push: use the event stream for continuous monitoring and `get_status` for an immediate, correlated read.

## When to Use

Use `get_status` when you need an **immediate, correlated snapshot of reader health** — power source, radio activity/connection, system clock, terminal link, and battery condition — typically in a management workflow over the MGMT endpoint. It solves the problem of confirming device state on demand rather than waiting for the next asynchronous heartbeat.

- **Get:** the current `deviceStatus` snapshot (power, radio, NTP, terminal connection, battery).
- **Use it for:** pre-flight safety checks before a state-changing command, spot health checks, and diagnostics. In particular, read `deviceStatus.radioActivity` before a `reboot` or `set_operating_mode`: if it is `ACTIVE`, `reboot` is rejected with code `5` (inventory in progress) and `set_operating_mode` is rejected with code `11` (inventory in progress).
- **Prefer a different command when:** you want continuous monitoring rather than a single read — subscribe to [`heartbeatEVT`](/observability/heartbeat) instead, which pushes the same health domains asynchronously.

The full pattern is `<tenantId>/<topic>/<deviceSerialNumber>`, where `<topic>` is configured per endpoint. See [Topic hierarchy](/foundations/mqtt/topic-hierarchy) and the [Topic quick reference](/reference/appendices/topic-quick-reference).

:::note
There is no per-message authentication block — credentials are presented once at CONNECT. The `requestId` you send is echoed in the response so you can match reply to request. QoS and retain are governed by the endpoint's `mqttParams`, not by this command.
:::

## Key Fields

- **`requestId`** (request) — your correlation token; make it unique per request. The device echoes it in the response so you can pair reply to request.
- **`deviceStatus.radioActivity`** (response) — `ACTIVE` or `INACTIVE`. Check it before `reboot` / `set_operating_mode`, which are rejected while the radio is `ACTIVE`.
- **`deviceStatus.powerSource`** (response) — how the reader is currently powered (`DC` / `WALLCHARGER` / `USB` / `CRADLE`).
- **`response.code`** (response) — read this to determine success vs failure (see [Response Codes](#response-codes)).

## Response payload

The response envelope is `command`, `requestId`, `apiVersion`, `response`, and an optional `deviceStatus` snapshot. On the **Schema** tab, expand `deviceStatus` (and its `ntp`, `terminalConnection`, `batteryStatus` sub-objects) to see every field — its type, whether it is required, its enum/units, an example value, and a description; the **Example** tab shows the full JSON. `deviceStatus` is optional at the envelope level; when present, its required members are `powerSource`, `radioActivity`, `radioConnection`, and `systemTime`. Optional members (including `hostname` and the top-level `temperature`) may be absent depending on device state and model.

## Response Codes

`get_status` returns `response.code` per `response.yaml`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success — status retrieved and returned (the only code observed live). |
| `3` | Not able to retrieve information — the reader could not gather the requested status at this moment. Retry; if persistent, reboot after stopping any active inventory. |
| `101` | Default error — generic catch-all. |

## Notes & Caveats

:::tip
Call `get_status` and inspect `deviceStatus.radioActivity` before a state-changing command. While it is `ACTIVE`, [`reboot`](/reference/mgmt/reboot) is rejected with code `5` and [`set_operating_mode`](/reference/ctrl/set-operating-mode) is rejected with code `11`. Stop inventory first — see [Start/stop inventory](/rfid/start-stop-inventory) and [Inventory control](/reference/ctrl/control-operation).
:::

:::note
The optional `deviceStatus` members (`hostname`, top-level `temperature`, `ntp`, `terminalConnection`, `batteryStatus`) may be absent depending on device state and model — this is normal behavior, not an error. `deviceStatus` reports two distinct temperatures: `deviceStatus.temperature` (device) and `deviceStatus.batteryStatus.temperature` (battery). Do not conflate them.
:::

:::info
The reader has no RTC backup battery. After a cold start or factory reset, before SNTP has synced, `systemTime` is a baseline default and `ntp.reach` is `0`. Time-sensitive operations (TLS certificate validity, log/event timestamps) may use the default time until `ntp.reach` becomes non-zero.
:::

:::note
`get_status` is the on-demand poll; the same health domains are also pushed asynchronously via [`heartbeatEVT`](/observability/heartbeat) and related alerts. Prefer the events for continuous monitoring and `get_status` for a point-in-time check. Cross-command `apiVersion` reporting can differ (for example [`mqttConnEVT`](/observability/mqtt-connection) emits a numeric `apiVersion`); that is a firmware note, not a `get_status` schema difference.
:::
