## Overview

`get_wifi` retrieves the **Wi-Fi profiles saved on the reader** together with their current connection status and configuration. The response is a list (`wifiProfiles.wifiConfig[]`) of profiles; each describes the wireless interface, the associated access point (ESSID, security type, connection status), the IPv4/IPv6 addressing, and a `securityDetails` block keyed by the security type (`WPA2Personal`, `WPA3Personal`, `WPA2Enterprise`, `WPA3Enterprise`). It is a read-only query on the device-management (MGMT) endpoint.

Stored credentials are **masked** by the device â€” passwords, passphrases, and identities are returned as a mask, never the real secret. Use `get_wifi` to verify which SSID a reader is connected to and confirm its addressing before relying on the wireless uplink.

## When to Use

Use `get_wifi` when you need to **verify the reader's wireless configuration** â€” during commissioning, a connectivity check before inventory, or a fleet audit of which SSID each device is on.

- **Confirm the connection:** check `accessPoint.status` (CONNECTED?), `accessPoint.essid` (correct network?), and `accessPoint.isPreferred`.
- **Validate security:** check `accessPoint.securityType` matches the AP; the matching `securityDetails` block shows the (masked) credential/protocol configuration.
- **Validate addressing:** check `ipv4Configuration.ipAddress`, `gateway`, and `enableDhcp` to confirm the reader has a working route to the broker / back-end.

The response envelope is `command`, `requestId`, `apiVersion`, `response`, and a `wifiProfiles` object whose `wifiConfig` is the **array of profiles**. On the **Schema** tab, expand `wifiProfiles` â†’ `wifiConfig` â†’ `interfaceDetails` (and its `accessPoint`, `ipv4Configuration`, `ipv6Configuration`, `securityDetails` sub-objects) to see every field. ## Notes & Caveats

:::warning
**Credentials are masked.** The device returns stored passwords, passphrases, and identities as a mask (never the real secret). Treat every credential field in the response as opaque â€” you cannot read back the actual value with `get_wifi`.
:::

:::note
**Security type selects the credential block.** `securityDetails` contains only the one key that matches `accessPoint.securityType` â€” `WPA2Personal` / `WPA3Personal` (a single `password`) or `WPA2Enterprise` / `WPA3Enterprise` (EAP `authentication`, optional `innerAuthentication` / `identity` / `anonymousIdentity`, a `certificate` array, and a `protocol`). `ipv6Configuration` is schema-defined but is not returned by the current firmware.
:::

:::warning
**Wi-Fi-disabled response shape (firmware quirk).** On the current firmware, when Wi-Fi is disabled the payload may be returned under a **top-level `wifiConfig`** object (rather than `wifiProfiles.wifiConfig[]`) and the interface `status` may be Title-cased (e.g. `Disabled` instead of the canonical `DISABLED`). This is a known firmware inconsistency pending a fix; the canonical (connected) shape documented here is `wifiProfiles.wifiConfig[]` with enum-cased status values.
:::

:::warning
**`apiVersion` is per-handler, not device-wide.** `get_wifi` reports `apiVersion` `V1.1` (within its documented enum `V1.0` / `V1.1`), but on the same firmware other commands (e.g. [`get_version`](/reference/mgmt/get-version), [`get_status`](/reference/mgmt/get-status)) report `V1.21`. The value is emitted by each command's handler, so do not treat it as a single device-wide constant.
:::
