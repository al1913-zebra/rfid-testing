## Overview

`reboot` performs a **warm reset** of the reader. It takes no payload beyond `command` and `requestId`. The device **accepts** the request (`response.code 1`), reboots, and then **re-initiates its connection to the previously-connected server on its own** — no manual reconnect needed. It is a state-changing command on the device-management (MGMT) endpoint.

:::danger
`reboot` is **session-severing**. The warm reset drops the MQTT control session for the duration of the reboot (observed **~30 s to ~2 min**, and it **varies** run-to-run). The reply is **`response.code 1` ("Command payload is accepted")**, returned **before** the reset — there is **no later "reboot complete" response**. Treat `reboot` as a deliberate, confirmed action: send it **once**, **never auto-retry**, and do not gate success on `code == 0`.
:::

## When to Use

Use `reboot` to **warm-reset the reader**:

- **Recover from a bad state** — clear a transient fault when a targeted command won't.
- **Apply/settle changes** that take effect on restart.
- **Clear the operating mode** — a reboot resets the operating mode to default (see the volatility note below).

Do **not** reboot during an active RFID inventory — the device refuses with **code `5`**. Stop the inventory first with [`control_operation`](/reference/ctrl/control-operation).

## Key Fields

- **`command`** / **`requestId`** — the entire request; there is no reboot payload.
- **`response.code`** — `1` means *accepted* (the reset then proceeds); `5` means the reboot was refused because an inventory is running. See [Response Codes](#response-codes).

## Reboot behavior

A single `reboot` drives the device through a clean warm-reset cycle:

1. **Accept** — the device replies `response.code 1` ("Command payload is accepted"), *before* it resets. This is the success-ack — there is no later "reboot complete" message.
2. **Down** — the reader drops its MQTT connections and warm-resets. It is unreachable for the duration.
3. **Auto-reconnect** — the device re-attaches to the **previously-connected server on its own**, with no manual intervention. Recovery time **varies** (observed ~44 s and ~110 s across two runs — roughly ~30 s to ~2 min); don't hard-code a fixed wait.
4. **State after reset:**
   - **Endpoint configuration persists** — saved and active endpoints are restored/re-activated across the reset.
   - **Operating mode is volatile** — the reboot resets the operating mode to default (e.g. `BALANCED_PERFORMANCE`, any armed `accessOperations` cleared). See [`operating mode`](/reference/ctrl/set-operating-mode); this is the documented "on reboot the set configuration is lost" behavior.

## Response Codes

`reboot` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `1` | Command payload is accepted (`IOT_STATUS_CMD_PAYLOAD_ACCEPTED`) — **the reboot success-ack**, returned before the reset. The device then reboots and auto-reconnects. |
| `0` | Success — the generic success code; in practice `reboot` acks with `1`, not `0`. |
| `5` | Can't reboot device, inventory in progress (`IOT_ERROR_IN_REBOOT_INVENTORY_IN_PROGRESS`) — **unique to `reboot`**; the reset is refused while an RFID inventory is running. Stop the inventory with [`control_operation`](/reference/ctrl/control-operation), then retry. |
| `101` | Default error — generic catch-all. |

## Notes & Caveats

:::danger
**Session-severing — send once, confirm first.** The warm reset drops the control session; the device auto-reconnects to the previously-connected server, but recovery time **varies** (~30 s–2 min observed). Because a reboot cannot be un-done and interrupts service, issue it deliberately (ideally after operator confirmation) and **never auto-retry** it — a lost accept does not mean the reboot didn't happen.
:::

:::note
**`code 1`, not `0`.** `reboot`'s success-ack is `code 1` ("Command payload is accepted"), returned before the reset — most other commands return `0` on success. A consumer that treats `code == 0` as the reboot success condition will misread every reboot reply. There is no terminal completion code on the response topic; confirm the device is back with [`get_version`](/reference/mgmt/get-version) or [`get_status`](/reference/mgmt/get-status) after it reconnects.
:::

:::note
**Refused during an active inventory.** If an RFID inventory is running, `reboot` returns **code `5`** ("Can't reboot device, inventory in progress"). Stop the inventory with [`control_operation`](/reference/ctrl/control-operation) before rebooting.
:::

:::note
**Endpoint config persists; operating mode does not.** Across the reset, saved and active MQTT endpoints are restored and re-activated, but the **operating mode reverts to default** (any tuned profile or armed `accessOperations` is cleared). Re-apply the operating mode after a reboot if you were relying on a non-default configuration — see [`operating mode`](/reference/ctrl/set-operating-mode).
:::

:::note
**`apiVersion` is per-handler.** `reboot` reports `apiVersion` `V1.1` (its response enum is `V1.0` / `V1.1`), while on the same firmware [`get_version`](/reference/mgmt/get-version) reports `V1.21`. Do not treat it as a single device-wide constant.
:::

:::tip
Sequence a reboot as: (optionally) stop any inventory with [`control_operation`](/reference/ctrl/control-operation) → send `reboot` once → wait out the reconnect (poll, don't hard-code the delay) → confirm with [`get_version`](/reference/mgmt/get-version) / [`get_status`](/reference/mgmt/get-status). For the concept-level view, see [System operations](/infrastructure/system-operations) and [Connection lifecycle](/foundations/mqtt/connection-lifecycle).
:::
