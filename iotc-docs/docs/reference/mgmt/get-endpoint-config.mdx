## Overview

`get_endpoint_config` **reads back the reader's MQTT endpoint configuration** — a faithful view of what was provisioned with [`config_endpoint`](/reference/mgmt/config-endpoint). It has two modes, chosen by whether you send `endpointDetails.endpointName`:

- **All-endpoints (omit `endpointDetails`):** returns every **active** endpoint's full configuration under `endpointResponse.activeEndpoints.epConfig[]`, **plus** the names of **all saved** endpoints under `endpointResponse.savedEndpoints.epNames[]`.
- **Single-endpoint (send `endpointDetails.endpointName`):** returns just that one endpoint's configuration.

It is a read-only command on the device-management (MGMT) endpoint.

:::warning
The response **includes broker credentials** — `mqttParams.password` (and `username` where set) are returned in the endpoint configuration. Treat the output of `get_endpoint_config` as **sensitive**: redact credentials in logs, tickets, and screenshots, and move it only over secured channels. Every credential is masked as `********` on this page.
:::

## When to Use

Use `get_endpoint_config` to **inspect or audit** the reader's MQTT wiring:

- **Before `config_endpoint`** — review the current configuration so you don't overwrite an active endpoint unexpectedly.
- **Confirm the plumbing** — check `url` / `port`, `protocol`, `verificationType`, and the `publishTopics` / `subscribeTopics` a broker will see.
- **List saved endpoints** — read `savedEndpoints.epNames[]` to see everything saved on the device (a superset of the active endpoints) before activating or deleting one.
- **Audit routing & security** — review each endpoint's `eventConfiguration` and `securityParams`.

## Topic Mapping

| Direction | Topic | Note |
| --- | --- | --- |
| Request (publish by app) | `zebra/MGMT/clients/cmnd/{serial}` | On the management endpoint. The topic base is endpoint-configurable; the reference test device used an `MDM` base (`zebra/MDM/clients/cmnd/{serial}`). |
| Response (subscribe by app; published by device) | `zebra/MGMT/clients/resp/{serial}` | Correlated reply to the request above; the in-payload `requestId` links the twin topics. Test device: `zebra/MDM/clients/resp/{serial}`. |

The full pattern is `<tenantId>/<topic>/<deviceSerialNumber>`. The `publishTopics` / `subscribeTopics` inside the response are **base** topics (e.g. `MDM/clients/cmnd`); on-wire they are wrapped to `{tenantId}/{base}/{serial}`. See [Topic hierarchy](/foundations/mqtt/topic-hierarchy) and the [Topic quick reference](/reference/appendices/topic-quick-reference).

:::note
There is no per-message authentication block — credentials are presented once at CONNECT. The `requestId` you send is echoed in the response so you can match reply to request. QoS and retain are governed by the endpoint's `mqttParams`, not by this command.
:::

## Key Fields

- **`endpointDetails.endpointName`** (request) — omit for all endpoints; supply to target one (case-sensitive).
- **`activeEndpoints.epConfig[]`** — the full config of each **active** endpoint. Inactive (saved-but-not-activated) endpoints do **not** appear here.
- **`savedEndpoints.epNames[]`** — the names of **all** saved endpoints (active *and* inactive) — the active/saved distinction in one field.
- **`configuration.mqttParams.password`** — ⚠️ a broker credential is present in the response; see [Response payload](#response-payload) and the credential warning below.
- **`response.code`** — read this to determine success vs failure (see [Response Codes](#response-codes)).

## Event routing fields

Each endpoint's `configuration.eventConfiguration` selects which events that endpoint emits. The schema (`cfgEventPayload.yaml`) declares the full set below, **but only 8 of the boolean toggles are functional today**; the remaining 12 keys are accepted by the firmware and returned in the read-back but **have no effect yet** (reserved for a future release). Enabling a reserved toggle silently does nothing.

| Field | Type | Status | Description |
| --- | --- | --- | --- |
| `terminalConnection` | boolean | ✅ Supported | Terminal connect/disconnect events. |
| `firmwareUpdate` | boolean | ✅ Supported | Firmware-update events. |
| `network` | boolean | ✅ Supported | Network events. |
| `heartbeat` | boolean | ✅ Supported | Periodic heartbeat events. |
| `power` | boolean | ✅ Supported | Power-source events. |
| `battery` | boolean | ✅ Supported | Battery events. |
| `temperature` | boolean | ✅ Supported | Temperature events. |
| `fileDownload` | boolean | ✅ Supported | File-download events. |
| `antenna` | boolean | 🔒 Reserved | Not yet functional. |
| `gpi` | boolean | 🔒 Reserved | Not yet functional. |
| `exceptions` | boolean | 🔒 Reserved | Not yet functional. |
| `ntp` | boolean | 🔒 Reserved | Not yet functional. |
| `userApp` | boolean | 🔒 Reserved | Not yet functional. |
| `cpuUsage` | boolean | 🔒 Reserved | Not yet functional. |
| `flashUsage` | boolean | 🔒 Reserved | Not yet functional. |
| `ramUsage` | boolean | 🔒 Reserved | Not yet functional. |
| `cpuThreshold` | integer | 🔒 Reserved | Not yet functional (paired with `cpuUsage`). |
| `ramThreshold` | integer | 🔒 Reserved | Not yet functional (paired with `ramUsage`). |
| `flashThreshold` | integer | 🔒 Reserved | Not yet functional (paired with `flashUsage`). |

The **8 supported** booleans are exactly the ones you can rely on today; configure event routing with [`config_events`](/reference/mgmt/config-events).

## Response Codes

`get_endpoint_config` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success — the endpoint configuration was retrieved. |
| `3` | Not able to retrieve information (`IOT_ERROR_INFO_NOT_AVAILABLE`) — the device could not gather the configuration at this time. Retry after a short delay; if persistent, reboot. |
| `101` | Default error — generic catch-all. |

## Examples

Publish `get_endpoint_config`; the device replies with the envelope shown under [Response payload](#response-payload). The all-endpoints request (`command` + `requestId`) returns every active endpoint plus the saved-names list; the single-endpoint request (shown under [Request payload](#request-payload)) targets one endpoint by name.

**Active vs saved.** In the response example, `activeEndpoints.epConfig[]` carries the full config of the one active endpoint (`mdm_ep`), while `savedEndpoints.epNames` lists all four saved (`mdm_ep`, `ctrl_ep`, `data1_ep`, `mgmt_ep`) — the three that are saved but not active appear **only** as names, with no config block. When more than one endpoint is active, `epConfig[]` holds one full configuration per active endpoint.

## Notes & Caveats

:::warning
**The response returns broker credentials.** `configuration.mqttParams.password` (and `username`, when set) are part of the read-back. Redact them in logs, tickets, and screenshots; transport and store the response only over secured channels. This page masks every credential as `********`.
:::

:::note
**Active vs saved.** `activeEndpoints.epConfig[]` contains a full configuration only for endpoints with `activate: true`. `savedEndpoints.epNames[]` lists **every** saved endpoint — so a name that appears in `epNames` but not in `epConfig[]` is saved but inactive.
:::

:::note
**`verificationType` — the device returns `VERIFY_NONE`.** A plain-MQTT (non-TLS) endpoint reads back `verificationType: "VERIFY_NONE"`. That value is **not yet in the schema enum** (`NONE`, `VERIFY_PEER`, `VERIFY_HOST`, `VERIFY_HOST_PEER`) — a known firmware/schema mismatch pending reconciliation (the device uses `NONE` when *setting* and reports `VERIFY_NONE` on *get*). Treat `VERIFY_NONE` as "no TLS verification."
:::

:::note
**`eventConfiguration` — 8 supported, 12 reserved.** Only the 8 supported booleans in [Event routing fields](#event-routing-fields) take effect today; the other 12 keys are returned in the read-back but have no effect yet. `heartbeatConfiguration` is returned only after [`config_events`](/reference/mgmt/config-events) enables the heartbeat.
:::

:::note
**Topic items and the `protocol` spelling.** Each `publishTopics` / `subscribeTopics` entry carries `topic`, `qos`, and `retain`; the schema formally requires only `qos` and `retain` per item (the device always supplies `topic`). The TLS protocol enum value is spelled `MQT-TLS` in the schema (a likely typo for `MQTT_TLS`); use whichever value a `get_endpoint_config` read-back reports for your device.
:::

:::note
**`apiVersion` is per-handler.** `get_endpoint_config` reports `apiVersion` `V1.1`; on the same firmware other commands (e.g. [`get_version`](/reference/mgmt/get-version)) report `V1.21`. Do not treat it as a single device-wide constant.
:::

:::tip
Read `get_endpoint_config` **before** [`config_endpoint`](/reference/mgmt/config-endpoint) to review the current wiring and avoid clobbering an active endpoint. For the concept-level view, see [How the MQTT plumbing fits together](/infrastructure/mqtt-endpoints).
:::
