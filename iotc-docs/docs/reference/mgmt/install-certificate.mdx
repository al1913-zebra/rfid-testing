> 📕 **REFERENCE** · **Audience:** API consumer · **Use:** install a certificate or key into the reader's trust store

## Overview

`install_certificate` **installs a certificate (or key) into the reader's trust store** — for MQTT-TLS brokers, Wi-Fi Enterprise (EAP), the file store, or client/server auth. Content is supplied one of two ways, chosen by `certDetails.certSource`:

- **`HTTP`** (default) — the device **downloads** each component from the URLs in `url[]`.
- **`DIRECT`** — you provide the content **inline** in `certificateBundle`.

It is a state-changing command on the device-management (MGMT) endpoint. Installed certificates are then referenced by name from [`config_endpoint`](/reference/mgmt/config-endpoint) (`MQTT_TLS`) and [`set_wifi`](/reference/mgmt/set-wifi) (Enterprise).

:::danger
`install_certificate` is **asynchronous**. A successful reply is **`response.code 1` ("Command payload is accepted")** — an *acknowledgement that the command was received*, **not** confirmation that the certificate was installed. There is **no terminal success/failure result on the response topic**. You must confirm completion out-of-band by reading back with [`get_installed_certificate`](/reference/mgmt/get-installed-certificate) (and, for `HTTP`, by watching the `FILE_DOWNLOAD` alerts on the event topic). Do **not** gate success on `code == 0`.
:::

## When to Use

Use `install_certificate` to **provision the certificates other commands depend on**:

- **Before `MQTT_TLS`** — install the `ca_cert` / `client_cert` / `client_key` (type `mqtt`) that [`config_endpoint`](/reference/mgmt/config-endpoint) references in `securityParams`.
- **Before Wi-Fi Enterprise** — install `wifi`-type certificates that [`set_wifi`](/reference/mgmt/set-wifi) references in its `certificate` array.
- **File-store trust** — install a `filestore` CA used to authenticate HTTPS downloads.
- **Choose the source:** use **`DIRECT`** when you can embed the content (no network dependency); use **`HTTP`** when the device can reach your file server (watch the `FILE_DOWNLOAD` alerts).

## Key Fields

- **`certDetails.certSource`** — `HTTP` (use `url`) or `DIRECT` (use `certificateBundle`).
- **`certDetails.type`** — the certificate category (`mqtt`, `wifi`, `filestore`, …); scopes where the certificate can be used.
- **`certDetails.authenticationType`** — `NONE` or `CERTIFICATE` (certificate-authenticated download).
- **`certificateBundle.client_key` / `cert_key_password`** — ⚠️ secrets you send inline (DIRECT); mask them. In `HTTP` mode a `url` entry with `key: cert_key_password` likewise carries a plaintext password in its `value`. See the credential warning below.
- **`response.code`** — `1` means *accepted*, not *installed*; confirm with [`get_installed_certificate`](/reference/mgmt/get-installed-certificate). See [Response Codes](#response-codes).

## Request payload

The envelope is `command`, `requestId`, and the `certDetails` payload. Use the **Schema** tab for the full field reference and the **Example** tab for an `HTTP` install of an `mqtt` certificate set.

:::warning
When you use `DIRECT`, `certificateBundle.client_key` is a **private key** and `cert_key_password` is a **password** — mask both as `********` in anything shared. In `HTTP` mode, a `url` entry with `key: cert_key_password` carries the key password as a plaintext `value` — mask that too. `install_certificate` is the one command where you transmit private-key material; the device never echoes it back (and [`get_installed_certificate`](/reference/mgmt/get-installed-certificate) never lists private keys).
:::

## Certificate sources

- **`HTTP` (default).** Supply `url[]` with a `{key, value}` entry per component (`ca_cert`, `client_cert`, `client_key`, `cert_key_password`). The device downloads each; for `authenticationType: CERTIFICATE` over HTTPS, supply `caCertificateFileContent` (or rely on an installed filestore CA). Requires **device → file-server reachability**; missing `url` returns code `29`. The device emits `FILE_DOWNLOAD` alerts as it fetches.
- **`DIRECT`.** Supply `certificateBundle` with inline PEM content for one `type`. No network dependency; missing `certificateBundle` returns code `30`.

## Response payload

The immediate reply is an **accept** — `command`, `requestId`, `apiVersion`, and `response` with **`code 1`**. It confirms receipt, not installation.

## Confirming completion

Because the accept (`code 1`) is not a terminal result, confirm the install actually landed:

1. **Read back with [`get_installed_certificate`](/reference/mgmt/get-installed-certificate)** — the installed components appear in `installedCerts.certInfo[]` (named `{name}_{component}`). This is the authoritative confirmation. (Note: private keys are never listed, and serials read back truncated to 8 bytes.)
2. **For `HTTP`, watch `FILE_DOWNLOAD` alerts** on the event topic — per-file download/save progress. These confirm download + save only, not trust-store persistence; still verify with the read-back.

There is **no** terminal success/failure message on the response topic after the `code 1` accept.

## Response Codes

`install_certificate` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `1` | Command payload is accepted (`IOT_STATUS_CMD_PAYLOAD_ACCEPTED`) — **the success/accept code for this command.** The install proceeds asynchronously; verify with `get_installed_certificate`. |
| `0` | Success — terminal success; in practice `install_certificate` acks with `1`, and `0` is what the `get_installed_certificate` read-back returns. |
| `9` | File not found (`IOT_STATUS_FILE_NOT_FOUND`) — a referenced file/URL could not be found. |
| `21` | Certificate not found (`IOT_ERROR_CERT_NOT_FOUND`) — the named certificate is absent. |
| `23` | Invalid enum value (`IOT_ERROR_INVALID_ENUM`) — a field (e.g. `type`, `certSource`, `authenticationType`, `verificationType`) is outside its allowed set. |
| `29` | Url missing for HTTP source (`IOT_ERROR_URL_MISSING`) — `certSource: HTTP` (or defaulted) with no/empty `url`. |
| `30` | Certificate content missing for direct source (`IOT_ERROR_CERT_CONTENT_MISSING`) — `certSource: DIRECT` with no `certificateBundle`. |
| `101` | Default error — generic catch-all. |

## Notes & Caveats

:::danger
**`code 1` is an async accept, not an install confirmation.** There is no terminal success/failure on the response topic. Always verify the install with [`get_installed_certificate`](/reference/mgmt/get-installed-certificate) (and, for `HTTP`, the `FILE_DOWNLOAD` alerts). A consumer that treats `code == 0` as the success condition will misread every `install_certificate` reply.
:::

:::warning
**You transmit private keys.** In `DIRECT` mode `certificateBundle.client_key` (private key) and `cert_key_password` (password) are secrets in the request — mask them as `********`. In `HTTP` mode, a `url` entry with `key: cert_key_password` also carries a plaintext password in its `value` — treat it the same way. Send `install_certificate` only over secured channels. The device never echoes them, and the read-back never exposes private-key material.
:::

:::note
**`type` and `authenticationType` are required but not firmware-enforced.** The schema requires both within `certDetails`, but the device accepts a payload omitting them (still returns `code 1`) and installs a malformed entry (e.g. `type: null`). Always send both; don't rely on the device to reject their absence.
:::

:::note
**Source rules.** `HTTP` (the default when `certSource` is omitted) requires `url` — missing → code `29`, and the device must be able to reach the file server. `DIRECT` requires `certificateBundle` — missing → code `30`. Invalid enum values (e.g. `type`, `certSource`) return code `23`. The request object has no `additionalProperties: false`, but the device rejects **unknown/extra fields** at parse time with code `2`.
:::

:::note
**`apiVersion` is per-handler.** `install_certificate` reports `apiVersion` `V1.21`, while on the same firmware [`get_installed_certificate`](/reference/mgmt/get-installed-certificate) reports `V1.1`. Do not treat it as a single device-wide constant.
:::

:::tip
Installing a certificate does **not** disturb the active MDM/MGMT control session. After installing, reference the certificate by its logical `name` from [`config_endpoint`](/reference/mgmt/config-endpoint) or [`set_wifi`](/reference/mgmt/set-wifi). For the concept-level view, see [TLS & certificates](/infrastructure/tls-and-certificates) and [Certificate management](/infrastructure/certificate-management).
:::
