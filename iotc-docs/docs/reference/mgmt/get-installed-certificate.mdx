> 📕 **REFERENCE** · **Audience:** API consumer · **Use:** list the certificates installed in the reader's trust store

## Overview

`get_installed_certificate` **lists the certificates in the reader's trust store**, returning per-certificate **metadata** — name, type, issuer/subject, serial, public-key algorithm, and validity dates — under `installedCerts.certInfo[]`. It is read-only, mutates nothing, and returns **no private-key material**. Use it to audit the certificate inventory and to confirm a certificate exists (and is valid) before referencing it from [`config_endpoint`](/reference/mgmt/config-endpoint) or [`set_wifi`](/reference/mgmt/set-wifi).

:::warning
**The wire command is the plural `get_installed_certificates`.** Although this operation is named `get_installed_certificate` (singular) — matching the schema file and the error-code mapping — the value you put in the `command` field must be the **plural `get_installed_certificates`**. The singular string is **not dispatched** by the firmware and produces **no reply**. Every example on this page uses the plural.
:::

## When to Use

Use `get_installed_certificate` to **audit or verify the reader's certificate inventory**:

- **Before configuring TLS** — confirm the `ca_cert` / `client_cert` referenced by [`config_endpoint`](/reference/mgmt/config-endpoint) (`MQTT_TLS`) or [`set_wifi`](/reference/mgmt/set-wifi) (Enterprise) is installed and of the right `type`.
- **Rotation planning** — check `validTill` (expiry) and `validFrom` (not-yet-valid) to schedule renewals before TLS handshakes start failing.
- **Trust audit** — review `issuerName` / `subjectName` / `serial` to confirm the expected CA chain.

## Key Fields

- **`certInfo[].type`** — the certificate's scope (`mqtt`, `wifi`, `filestore`, `client`, `server`); a certificate installed for one type is not usable by another.
- **`certInfo[].name`** — the logical name other commands reference (e.g. `ca_cert`, `client_cert`).
- **`certInfo[].validTill` / `validFrom`** — the validity window; an expired or not-yet-valid certificate causes TLS handshake failures.
- **`certInfo[].issuerName` / `subjectName` / `serial`** — identify the certificate and its CA chain (note `serial` is truncated — see Notes).
- **`response.code`** — read this to determine success vs failure (see [Response Codes](#response-codes)).

## Response Codes

`get_installed_certificate` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success — the inventory was retrieved (an empty trust store still returns `0` with an empty/absent `certInfo`). |
| `3` | Not able to retrieve information (`IOT_ERROR_INFO_NOT_AVAILABLE`) — the device could not gather the certificate list at this time. Retry after a short delay; if persistent, reboot. |
| `9` | File not found (`IOT_STATUS_FILE_NOT_FOUND`) — a backing certificate file could not be found. |
| `21` | Certificate not found (`IOT_ERROR_CERT_NOT_FOUND`) — the requested certificate is not installed. |
| `101` | Default error — generic catch-all. |

:::note
In live testing only `0` was observed for a normal read; codes `3` / `9` / `21` are this command's mapped failure codes per `error_codes.json`. Separately, a **malformed request** — including an **unknown/extra field** — is rejected with code `2` ("Invalid payload", a JSON-parser "No such field" error), even though the request schema does not declare `additionalProperties: false`. Send exactly `command` + `requestId`.
:::

## Notes & Caveats

:::warning
**Use the plural command literal.** Send `command: "get_installed_certificates"`. The singular `get_installed_certificate` (the operation/file name) is **not dispatched** — publishing it yields no device reply. An unknown or bogus command value is likewise silently dropped (no error), so a missing reply usually means a wrong command string.
:::

:::note
**`serial` is truncated to 8 bytes.** The device returns the certificate serial as colon-delimited hex, but only the **first 8 bytes** of the (typically 20-byte) real serial — e.g. `7d:ef:d4:c3:2d:39:4a:0b`. Do not rely on `serial` alone as a globally unique identifier; combine it with `name` + `type` + `issuerName`.
:::

:::note
**Private keys are never listed.** The inventory returns public certificate metadata only. A `client_key` installed via [`install_certificate`](/reference/mgmt/install-certificate) does **not** appear in `certInfo[]` — only its paired `ca_cert` / `client_cert` do. There is no way (by design) to read back private-key material.
:::

:::note
**`publicKey` and `publicKeyAlg` quirks.** The device currently populates `publicKey` with the algorithm name (e.g. `RSA`) rather than actual key bytes — treat `publicKey` as informational, not as usable key material. Also, the schema's `publicKeyAlg` enum has a stray trailing-quote typo on the SHA variants (`RSA_SHA_256"`, `RSA_SHA_384"`, `RSA_SHA_512"`); the intended values are `RSA_SHA_256` / `RSA_SHA_384` / `RSA_SHA_512`.
:::

:::note
**`apiVersion` is per-handler.** `get_installed_certificate` reports `apiVersion` `V1.1`, while on the same firmware [`install_certificate`](/reference/mgmt/install-certificate) and [`get_version`](/reference/mgmt/get-version) report `V1.21` (the response enum allows `V1.0` / `V1.1` / `V1.21`). Do not treat it as a single device-wide constant.
:::

:::tip
**Read-only, but still sensitive.** The inventory exposes no secrets (no private keys), but a certificate list reveals the device's PKI/trust topology — redact it in shared logs, tickets, and screenshots. For the concept-level view, see [TLS & certificates](/infrastructure/tls-and-certificates) and [Certificate management](/infrastructure/certificate-management).
:::
