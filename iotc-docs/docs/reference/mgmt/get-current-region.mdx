## Overview

`get_current_region` retrieves the reader's **active regulatory / RF region configuration** in a single round trip: the country and governing regulatory standard, the channel plan (the frequencies the radio may use), the transmit-power limits, and whether Listen-Before-Talk (LBT) and frequency hopping are engaged. It is a read-only query on the device-management (MGMT) endpoint.

The region governs whether ‚Äî and how ‚Äî the reader is legally allowed to transmit, so this read is the **compliance gate before any radio operation**. Note that the region can only be **read** over MQTT; it is **set** with 123RFID Desktop at bootstrap (see [What your reader knows about itself](/infrastructure/device-state)).

## When to Use

Use `get_current_region` when you need to **confirm the reader's regulatory configuration before it transmits** ‚Äî typically in provisioning verification, a pre-inventory check, or a fleet compliance audit.

- **Confirm the region:** verify `country` and `regulatoryStandard` match the deployment location before starting inventory.
- **Validate the RF envelope:** check `maxTxPowerSupported` / `minTxPowerSupported`, `channelData`, `lbtEnabled`, and `frequencyHopping` against the constraints your operation expects.
- **Audit compliance across a fleet:** read the region on each device to confirm consistent, compliant settings.
- **After provisioning or a region change:** run `get_current_region` to confirm the new settings took effect before relying on the radio.

## Key Fields

- **`requestId`** (request) ‚Äî your correlation token; make it unique per request. The device echoes it in the response so you can pair reply to request.
- **`currentRegion.country`** / **`regulatoryStandard`** (response) ‚Äî the configured country and governing standard (e.g. `FCC`, `ETSI`); confirm they match the deployment location.
- **`currentRegion.maxTxPowerSupported`** (response) ‚Äî the power ceiling (dBm); validate transmit-power settings against it before inventory.
- **`currentRegion.lbtEnabled`** / **`frequencyHopping`** (response) ‚Äî compliance behaviors that affect how (and whether) inventory can start.
- **`response.code`** (response) ‚Äî read this to determine success vs failure (see [Response Codes](#response-codes)).

## Response Codes

`get_current_region` returns `response.code` per `response.yaml` and `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success ‚Äî region configuration retrieved and returned (the only code observed live). |
| `3` | Not able to retrieve information (`IOT_ERROR_INFO_NOT_AVAILABLE`) ‚Äî the device could not gather the region information at this time. Retry after a short delay; if it persists, reboot the device. |
| `101` | Default error ‚Äî generic catch-all. |

:::note
An **unconfigured region** surfaces as code `6` (`IOT_STATUS_REGION_NOT_CONFIGURED`) at **connect / bootstrap time** ‚Äî it is not returned by `get_current_region` itself. If a device has no region set, configure it with 123RFID Desktop before connecting.
:::

## Notes & Caveats

:::warning
**The region is read-only over MQTT.** It is **set** with 123RFID Desktop at bootstrap, not by any MQTT command ‚Äî `get_current_region` only reads it. A reader **will not transmit at all until a region is set**, and a reader set to the wrong region will operate but may violate local regulations. If a freshly provisioned reader reports the wrong `country`, redo region setup in 123RFID Desktop. See [What your reader knows about itself](/infrastructure/device-state) and [Regulatory information](/reference/appendices/regulatory).
:::

:::note
`channelData` is a list of channel **center frequencies in kHz**, as strings ‚Äî e.g. `"865700"` = 865.7 MHz. The count and values depend on the configured region: US / FCC has ~50 channels across 902‚Äì928 MHz; ETSI regions typically have fewer, in the 865‚Äì868 MHz band. `country` and `regulatoryStandard` are free-form, device-driven strings (no fixed enum).
:::

:::info
`maxTxPowerSupported` / `minTxPowerSupported` are documented in **dBm** (e.g. `30` dBm ‚âà 1 W). If precise transmit power matters to your integration, confirm the scale against your firmware, as some readings may be reported in deci-dBm rather than literal dBm.
:::

:::warning
**`apiVersion` is per-handler, not device-wide.** `get_current_region` reports `apiVersion` `V1.1` (within its documented enum `V1.0` / `V1.1`), but on the same firmware other commands (e.g. [`get_version`](/reference/mgmt/get-version), [`get_status`](/reference/mgmt/get-status)) report `V1.21`. The value is emitted by each command's handler, so do not treat it as a single device-wide constant.
:::

:::tip
Run `get_current_region` **after device provisioning or a region change** to confirm the regulatory settings are correct before starting inventory operations.
:::
