> 📕 **REFERENCE** · **Audience:** API consumer · **Use:** add, update, or delete an MQTT endpoint configuration on the reader

## Overview

`config_endpoint` **provisions the reader's MQTT endpoints** — it **adds**, **updates**, or **deletes** an endpoint configuration (chosen by `epConfig.operation`). One payload defines an endpoint's type, protocol, broker `url`/`port`, TLS `verificationType` and certificates, MQTT `keepAlive`/session/reconnect settings, publish/subscribe topics, optional broker credentials, and per-endpoint event routing. It is a state-changing command on the device-management (MGMT) endpoint; the response is **status-only**. Read the result back with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config).

:::warning
`config_endpoint` is **state-changing** and **can carry broker credentials** (`mqttParams.username` / `password`). Mask credentials as `********` in anything shared. Take special care when reconfiguring the endpoint that carries your **own control session** — see the activation notes below. A saved endpoint is reversible with `operation: delete`.
:::

## Details

| Property | Value |
| --- | --- |
| **Pattern Name** | Request / Response (app publishes a command; device publishes the correlated reply) |
| **Applies To** | Device-management (MGMT) endpoint; RFD40 series, RFD90 series (verified on RFD40) |
| **Related Commands** | [`get_endpoint_config`](/reference/mgmt/get-endpoint-config), [`config_events`](/reference/mgmt/config-events), [`install_certificate`](/reference/mgmt/install-certificate), [`reboot`](/reference/mgmt/reboot) |
| **Required Request Fields** | `command`, `requestId` (plus `epConfig` with `operation` + `configuration`) |
| **Supported Operations** | `add` (new) · `update` (existing) · `delete` (existing) |
| **Endpoint Types** | `MGMT`, `MGMT_EVT`, `CTRL`, `DATA1`, `DATA2`, `SOTI`, `MDM` |
| **Protocols** | `MQTT` (plain) · `MQTT_TLS` (TLS) |
| **State Change** | STATE-CHANGING (writes/removes a saved endpoint; may activate it) |
| **Reversibility** | REVERSIBLE — remove with `operation: delete`; change with `operation: update` |

## When to Use

Use `config_endpoint` to **provision or maintain the reader's MQTT wiring** after the initial `MDM` endpoint exists (that first endpoint is set up with 123RFID Desktop; all others are added remotely with `config_endpoint`).

- **`add`** — create a new endpoint (`MGMT`, `CTRL`, `DATA1`, …). The `endpointName` must not already exist (duplicate → code `10`).
- **`update`** — modify an existing endpoint (e.g. change `url`, or flip `activate`). The `endpointName` must already exist.
- **`delete`** — remove an existing endpoint (only `endpointName` + `epType` needed).
- **Pre-provision safely** — send `activate: false` to save an endpoint without activating it, then activate later with `update`.

## Topic Mapping

| Direction | Topic | Note |
| --- | --- | --- |
| Request (publish by app) | `zebra/MGMT/clients/cmnd/{serial}` | On the management endpoint. The topic base is endpoint-configurable; the reference test device used an `MDM` base (`zebra/MDM/clients/cmnd/{serial}`). |
| Response (subscribe by app; published by device) | `zebra/MGMT/clients/resp/{serial}` | Correlated reply; the in-payload `requestId` links the twin topics. Test device: `zebra/MDM/clients/resp/{serial}`. |

The `topic` values you configure inside `mqttParams` are **base** topics (e.g. `MGMT/clients/resp`). The reader constructs the full topic at runtime as `<tenantId>/<topic>/<deviceSerialNumber>` — never include the tenant or serial in `topic`. See [Topic hierarchy](/foundations/mqtt/topic-hierarchy) and the [Topic quick reference](/reference/appendices/topic-quick-reference).

:::note
There is no per-message authentication block — credentials are presented once at CONNECT (and, for a configured endpoint, via `mqttParams`). The `requestId` you send is echoed in the response so you can match reply to request.
:::

## Key Fields

- **`epConfig.operation`** — `add` / `update` / `delete`; determines the action.
- **`configuration.endpointName`** — the endpoint's identity (create target / update / delete key).
- **`configuration.epType`** — the endpoint's role (`MGMT`, `CTRL`, `DATA1`, …).
- **`configuration.activate`** — `true` activates immediately; `false` saves without activating.
- **`configuration.securityParams`** — for `MQTT_TLS`; the referenced certificates must be pre-installed via [`install_certificate`](/reference/mgmt/install-certificate).
- **`response.code`** — read this to determine success vs failure (see [Response Codes](#response-codes)).

## Operations

`epConfig.operation` selects the action. The schema lists all 9 `configuration` fields as required (written for the `add` case); `delete` and `update` accept a **partial** `configuration`:

- **`add`** — creates a new endpoint. Supply the full `configuration` (all 9 required fields, plus optional `mqttParams` / `securityParams` / `eventConfiguration`). Fails with code `10` if the `endpointName` already exists.
- **`update`** — modifies an existing endpoint (matched by `endpointName`). Send `endpointName` (+`epType`) plus the fields you are changing.
- **`delete`** — removes an existing endpoint. Only `endpointName` and `epType` are needed.

:::note
`add` is **not idempotent.** If a response is lost (e.g. a timeout), the endpoint may already have been created — **read back with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config) before retrying** rather than blindly re-sending, or the retry will be a duplicate `add`.
:::

## Response payload

`config_endpoint` returns a **status-only** envelope — `command`, `requestId`, `apiVersion`, and `response`. There is no configuration echo; confirm the change with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config).

## Response Codes

`config_endpoint` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success — the endpoint was added, updated, or deleted. |
| `10` | Configuration already exist (`IOT_ERROR_CONFIG_ALREADY_EXIST`) — an `add` used an `endpointName` that already exists. Use `update`, or delete it first. |
| `23` | Invalid enum value (`IOT_ERROR_INVALID_ENUM`) — a field (e.g. `operation`, `epType`, `protocol`, `verificationType`, `format`) is outside its allowed set. |
| `25` | Max 3 publish topics exceeded (`IOT_ERROR_PUBLISH_TOPICS_EXCEEDED`) — more than 3 `publishTopics`. |
| `26` | Max 1 subscribe topic exceeded (`IOT_ERROR_SUBSCRIBE_TOPIC_EXCEEDED`) — more than 1 `subscribeTopics`. |
| `27` | Invalid tenant ID length (`IOT_ERROR_INVALID_TENANTID_LENGTH`) — `tenantId` exceeds the allowed length. |
| `33` | Endpoint already configured (`IOT_ERROR_ENDPOINT_ALREADY_CONFIGURED`) — an endpoint with the same configuration already exists. |
| `101` | Default error — generic catch-all. See the duplicate-`add` note below. |

:::note
**Duplicate `add` may return the generic `101`.** The documented code for adding an `endpointName` that already exists is `10` ("Configuration already exist"), but the firmware has been observed returning the generic `101` ("Default error") for a duplicate `add` instead of `10`. Treat a `101` after a re-sent `add` as a probable duplicate and verify with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config).
:::

## Notes & Caveats

:::warning
**Payloads can carry broker credentials.** `mqttParams.username` / `password` are supplied in the request when the broker requires them — masked here as `********`. Redact them in logs, tickets, and screenshots, and send `config_endpoint` only over secured channels. (Certificate fields — `caCertificateFile` / `clientCert` / `clientKey` — are **reference names** of already-installed certificates, not key material.)
:::

:::danger
**Reconfiguring the active control endpoint can disrupt your session.** Activating a *new* endpoint of a different role coexists with the endpoint carrying your control session (e.g. activating a `DATA1` endpoint does not drop the `MDM`/`MGMT` control channel). But changing the broker `url`/`port`/`protocol` of — or deleting — the endpoint that carries your **own** control session can sever it. When in doubt, save with `activate: false` and switch deliberately.
:::

:::note
**TLS needs pre-installed certificates.** For `protocol: MQTT_TLS`, the `securityParams` `caCertificateFile` / `clientCert` / `clientKey` must reference certificates already installed via [`install_certificate`](/reference/mgmt/install-certificate). Only `format: PEM` is supported today.
:::

:::note
**Topic limits.** `publishTopics` allows **up to 3** entries (exceeding → code `25`); `subscribeTopics` allows **up to 1** (exceeding → code `26`). Configure only the base `topic`; the reader wraps it as `<tenantId>/<topic>/<serial>`.
:::

:::note
**`verificationType` reads back differently.** You **send** a schema enum value (`NONE`, `VERIFY_PEER`, `VERIFY_HOST`, `VERIFY_HOST_PEER`), but [`get_endpoint_config`](/reference/mgmt/get-endpoint-config) reads back firmware-mapped tokens — e.g. `NONE` → `VERIFY_NONE`, `VERIFY_HOST_PEER` → `VERIFY_HOST_AND_PEER` — which are not in the schema enum (a known firmware/schema mismatch, pending reconciliation). Send the schema values.
:::

:::note
**Event routing.** `configuration.eventConfiguration` uses the same shape as [`config_events`](/reference/mgmt/config-events); only 8 of its toggles are functional today (the rest are reserved). See the Event routing fields table on the [`get_endpoint_config`](/reference/mgmt/get-endpoint-config) page for the full supported-vs-reserved split.
:::

:::note
**`apiVersion` is per-handler.** `config_endpoint` reports `apiVersion` `V1.1` (its response enum also allows `V1.0` / `V1.21`); on the same firmware other commands (e.g. [`get_version`](/reference/mgmt/get-version)) report `V1.21`. Do not treat it as a single device-wide constant.
:::

:::tip
Because the response is status-only, **verify the result with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config)** — confirm the endpoint appears in `savedEndpoints.epNames[]` and (if `activate: true`) under `activeEndpoints`. For the concept-level view, see [How the MQTT plumbing fits together](/infrastructure/mqtt-endpoints).
:::
