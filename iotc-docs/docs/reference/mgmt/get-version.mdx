## Overview

`get_version` retrieves the reader's **identity and software-version metadata** in a single round trip: model, serial number, SKU, top-level firmware version, and a component-level breakdown (scanner, radio, and IoT Connector versions). It is a read-only query on the device-management (MGMT) endpoint — the app publishes the command and the device publishes a correlated reply containing the `readerVersion` object.

Because it changes nothing, is idempotent, and does not depend on radio state, `get_version` is the natural **first command to send after any connect, reconnect, or bootstrap** — to confirm which device you are talking to and which command set it supports. Where `get_version` reports *identity*, [`get_status`](/reference/mgmt/get-status) reports live *runtime health*.

## When to Use

Use `get_version` when you need to **establish device identity and the software baseline** — typically in a management or fleet workflow over the MGMT endpoint. It answers "is this the device I expect, and what command set does it support?"

- **Identify the device:** `readerVersion.model`, `serialNumber`, and `sku` confirm the exact model and regional/hardware variant, and key asset-tracking, support tickets, and device registration to the physical label.
- **Gate on firmware / capabilities:** compare `readerVersion.firmwareVersion` and `detailedVersions` against your expected baseline before an update or when diagnosing an issue; read `detailedVersions.iotcVersion` to decide which MQTT API commands and features are available.
- **Confirm a firmware update applied:** [`set_os`](/reference/mgmt/set-os) returns only an *accepted* acknowledgement (`code 1`) with no terminal success/failure on the response topic — poll `get_version.readerVersion.firmwareVersion` to verify the new firmware actually took effect.

:::note
There is no per-message authentication block — credentials are presented once at CONNECT. The `requestId` you send is echoed in the response so you can match reply to request. QoS and retain are governed by the endpoint's `mqttParams`, not by this command.
:::

## Key Fields

- **`requestId`** (request) — your correlation token; make it unique per request. The device echoes it in the response so you can pair reply to request.
- **`readerVersion.model`** (response) — `RFD40` or `RFD90`; branch model-specific configuration on it.
- **`readerVersion.firmwareVersion`** (response) — the reader firmware build; compare against your baseline and use it to confirm a `set_os` update applied.
- **`readerVersion.detailedVersions.iotcVersion`** (response) — the IoTC version, which determines the available command/feature set.
- **`response.code`** (response) — read this to determine success vs failure (see [Response Codes](#response-codes)).

| Code | Description |
| --- | --- |

## Notes & Caveats

:::tip
`get_version` is idempotent and does not depend on radio state, so it is the right **first command after any connect, reconnect, or bootstrap** — confirm the device identity and IoTC version before relying on newer commands or fields. See [What your reader knows about itself](/infrastructure/device-state).
:::

:::warning
**`apiVersion` is per-handler, not device-wide.** On current firmware, `get_version` may report `apiVersion` `V1.21` while another command reports `V1.1` in the same session — the value is emitted by each command's handler, so do not treat it as a single device-wide constant. Use `readerVersion.detailedVersions.iotcVersion` as the IoTC software version (it reported `V1.21` here), and compare firmware builds via `readerVersion.firmwareVersion`.
:::

:::note
Confirming a firmware update: [`set_os`](/reference/mgmt/set-os) returns `code 1` ("Command payload is accepted") and delivers **no terminal success/failure** on the response topic. To confirm the update actually applied, poll `get_version` and check that `readerVersion.firmwareVersion` advanced to the expected build.
:::

:::note
`companyName` and `manufacturerName` are constrained to `Zebra Technologies`. `readerVersion.detailedVersions` is optional and may be absent on some builds; the example values shown (firmware, SKU, serial) are from a live RFD40 and are illustrative — real values vary by unit and region.
:::
