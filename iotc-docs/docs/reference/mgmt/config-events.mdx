> 📕 **REFERENCE** · **Audience:** API consumer · **Use:** enable/disable the reader's event streams and configure the heartbeat

## Overview

`config_events` **enables or disables which device-management events the reader emits**, sets the **heartbeat** interval and payload, and configures **alert thresholds**. You send a single `eventConfiguration` object of boolean flags (plus `heartbeatConfiguration` and thresholds). It is a state-changing **management-plane (MGMT) command** and carries no credentials.

Two things make this command distinctive:

- **Changes take effect after a reboot.** The configuration is **stored immediately** (visible via [`get_endpoint_config`](/reference/mgmt/get-endpoint-config)), but the reader only starts **emitting** the configured events after a [`reboot`](/reference/mgmt/reboot) — per the command's own schema description.
- **Accept-but-drop.** The reader accepts the whole payload with `code 0` even if it contains flags it doesn't support. Of the declared flags, **8 are functional** today; the rest are **declared for future support but not emitted on current V1.1 firmware** — most are dropped, though some (e.g. `ntp`) are stored and echoed back yet stay inert. Always **read back** with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config) to see what was stored — but note a stored *reserved* flag has still not taken functional effect.

:::warning[Changes apply on reboot]
`config_events` **stores** the configuration immediately, but the reader begins **emitting** the newly-enabled events only **after a [`reboot`](/reference/mgmt/reboot)** (the command schema states: *"To apply the changes, a device reboot is required."*). Don't assume events are live until the reader has rebooted.
:::

## When to Use

Use `config_events` to **control the reader's management-event surface**:

- **Turn event streams on/off** — enable `terminalConnection`, `firmwareUpdate`, `network`, `power`, `battery`, `temperature`, `fileDownload`, or `heartbeat`.
- **Shape the heartbeat** — set `heartbeatConfiguration.interval` and choose whether each heartbeat carries inventory / battery status.
- **Reduce noise** — disable streams you don't consume so the reader emits less telemetry.

Then **[`reboot`](/reference/mgmt/reboot)** to activate emission, and **[`get_endpoint_config`](/reference/mgmt/get-endpoint-config)** to confirm the stored configuration.

## Topic Mapping

`config_events` is a **management-plane command** — answered on the MGMT/MDM endpoint, not the control endpoint.

| Direction | Topic | Note |
| --- | --- | --- |
| Request (publish by app) | `zebra/MGMT/clients/cmnd/{serial}` (or `zebra/MDM/clients/cmnd/{serial}`) | On the management endpoint that carries your session. |
| Response (subscribe by app) | `zebra/MGMT/clients/resp/{serial}` (or `zebra/MDM/clients/resp/{serial}`) | Correlated reply; the in-payload `requestId` links the twin topics. |
| Configured events (subscribe by app) | `zebra/MGMT/clients/event/{serial}` (or `zebra/MDM/clients/event/{serial}`) | Where the enabled events flow, **after a reboot**. |

The full pattern is `<tenantId>/<topic>/<deviceSerialNumber>`. `config_events` configures the **event configuration of the management endpoint on which it is received** — other endpoints' event settings (set via [`config_endpoint`](/reference/mgmt/config-endpoint)) are left untouched. See [Topic hierarchy](/foundations/mqtt/topic-hierarchy) and the [Topic quick reference](/reference/appendices/topic-quick-reference).

## Supported vs reserved events

Every flag below is **declared** in the schema, but only a subset is functional on current V1.1 firmware. The reader **accepts** a reserved flag (returns `code 0`) but does **not** emit it — the API developer has declared these for **future support** and they should be treated as **not currently supported**.

| Event flag | Status | Stream |
| --- | --- | --- |
| `terminalConnection` | ✅ Supported | Terminal (cradle / USB-C) connect / disconnect |
| `firmwareUpdate` | ✅ Supported | Firmware-update progress / completion |
| `network` | ✅ Supported | Network interface / connectivity changes |
| `heartbeat` | ✅ Supported | Periodic heartbeat (shaped by `heartbeatConfiguration`) |
| `power` | ✅ Supported | Power-source change (battery / external) |
| `battery` | ✅ Supported | Battery state (low, critical, full) |
| `temperature` | ✅ Supported | Temperature alert |
| `fileDownload` | ✅ Supported | File-download progress / completion |
| `antenna` | ⚠️ Reserved | Not emitted on V1.1 |
| `gpi` | ⚠️ Reserved | Not emitted on V1.1 |
| `exceptions` | ⚠️ Reserved | Not emitted on V1.1 |
| `ntp` | ⚠️ Reserved | Not emitted on V1.1 (may be stored/echoed) |
| `userApp` | ⚠️ Reserved | Not emitted on V1.1 |
| `cpuUsage` | ⚠️ Reserved | Not emitted on V1.1 |
| `flashUsage` | ⚠️ Reserved | Not emitted on V1.1 |
| `ramUsage` | ⚠️ Reserved | Not emitted on V1.1 |

The four **threshold** fields (`cpuThreshold`, `ramThreshold`, `flashThreshold`, `temperatureThreshold`) and `heartbeatConfiguration.userApps` are likewise **reserved** — declared but not honored on V1.1. Enabling any reserved field is harmless (it does not error) but has no effect. See [The IOTC event model](/observability/events/model).

## Heartbeat configuration

When `heartbeat: true`, `heartbeatConfiguration` shapes each heartbeat message:

| Field | Type | Meaning |
| --- | --- | --- |
| `interval` | integer (seconds) | Seconds between heartbeats. Lower = more telemetry and higher battery cost. |
| `inventoryStatus` | boolean | Include the inventory-status block (`rfidStatus`, `tagCount`, `scanCount`). |
| `batteryStatus` | boolean | Include the battery-status block. |
| `userApps` | boolean | ⚠️ Reserved — not functional on V1.1. |

See [Watch your reader's pulse](/observability/heartbeat) for the heartbeat event shape.

## Response payload

:::note
Unlike the control-plane commands (`set_operating_mode`, `set_post_filter`, `control_operation`), which omit `response.code` on success, `config_events` is a management command and **includes `code: 0`** on success (live-verified). Both `command` and a `code`/`description` are always present.
:::

## Response Codes

`error_codes.json` lists **no command-specific error codes** for `config_events` — only the universal codes apply. In practice the command is highly permissive (it accepts unsupported flags with `code 0` rather than rejecting them; see Notes), so success is by far the common outcome.

| Code | Description |
| --- | --- |
| `0` | Success — the configuration was stored (`code` present, `description: "Success"`). Emission activates on the next reboot. |
| `101` | Default error — generic catch-all for an unhandled failure. |

A malformed request (e.g. a non-object `eventConfiguration`) could surface a generic invalid-payload error (`2`), but this command is not enumerated under any specific code in `error_codes.json` and no such error was observed in testing. For the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

## Notes & Caveats

:::note
**Stored immediately, emitted after a reboot.** `config_events` writes the configuration right away — you can confirm it with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config) before rebooting — but the reader only starts **emitting** the enabled events after a [`reboot`](/reference/mgmt/reboot). Plan a reboot into any event-configuration change.
:::

:::warning[Accept-but-drop — always read back]
The reader accepts the full payload with `code 0` even when it contains flags/thresholds it does not support. There is no `additionalProperties: false` on the payload schema, so nothing is rejected. It honors the **supported** subset; **reserved** flags are mostly dropped, though some (notably `ntp`) are stored and echoed back in the read-back while remaining **unemitted**. Enabling a reserved flag is harmless but has no effect. **Always read back with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config)** — but remember that a *reserved* flag appearing in the read-back is stored, not functional; only the 8 supported flags take effect.
:::

:::note
**Send the complete configuration — treat it as a wholesale set.** In testing, the stored `eventConfiguration` reflected exactly the supported fields submitted (a run that omitted `heartbeatConfiguration` and set `heartbeat: false` left **no** `heartbeatConfiguration` stored). Don't rely on omitted flags retaining their previous value — send every flag you want set, then verify with [`get_endpoint_config`](/reference/mgmt/get-endpoint-config).
:::

:::note
**8 supported flags; the rest are reserved.** Only `terminalConnection`, `firmwareUpdate`, `network`, `heartbeat`, `power`, `battery`, `temperature`, and `fileDownload` are emitted on current V1.1 firmware. `antenna`, `gpi`, `exceptions`, `ntp`, `userApp`, `cpuUsage`, `flashUsage`, `ramUsage`, the four `*Threshold` fields, and `heartbeatConfiguration.userApps` are **declared for future support but not functional** — see the [Supported vs reserved events](#supported-vs-reserved-events) table and [The IOTC event model](/observability/events/model).
:::

:::note
**`command` is not enum-constrained.** The command schema declares `command` with only an example (no `enum`), so a wrong string wouldn't be caught by schema validation. Always send exactly `config_events`.
:::

:::note
**Targets the receiving management endpoint.** `config_events` configures the `eventConfiguration` of the MGMT/MDM endpoint on which it is received; other endpoints' event settings (set via [`config_endpoint`](/reference/mgmt/config-endpoint)) are unchanged.
:::

:::note
**`apiVersion` is per-handler.** `config_events` reports `apiVersion` `V1.1` (its response enum is `V1.0` / `V1.1`), while on the same firmware [`get_version`](/reference/mgmt/get-version) reports `V1.21`. Do not treat it as a single device-wide constant.
:::

:::tip
`config_events` carries **no credentials** — only event flags and thresholds, so there is nothing to mask. It is **reversible**: re-send with the desired flags to restore a prior configuration (followed by a reboot). For the concept-level view, see [Choose what the reader tells you](/observability/configure-events) and [Event configuration](/observability/configure-events).
:::
