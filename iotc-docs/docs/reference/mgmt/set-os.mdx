> 📕 **REFERENCE** · **Audience:** API consumer · **Use:** initiate an over-the-air firmware update on the reader

## Overview

`set_os` **initiates an over-the-air (OTA) firmware update** — it points the reader at a firmware `.DAT` image at a URL and triggers the download-and-flash cycle. It is a state-changing command on the device-management (MGMT) endpoint, and it is both **asynchronous** and **session-severing**: the device accepts the payload, downloads and flashes the firmware, and **reboots** — dropping off the network for the duration. The only way to confirm the outcome is an out-of-band [`get_version`](/reference/mgmt/get-version) read-back.

:::danger
`set_os` **reboots and reflashes the reader.** Plan for the device to be unreachable during the update and to re-attach its MQTT session afterward. The reply is **`response.code 1` ("Command payload is accepted")** — an accept, **not** confirmation that the firmware updated (the same `code 1` is returned for both success and failure). And while the update runs, the device rejects **every other command** with **`code 4`** ("Firmware update in progress"). Confirm the result with [`get_version`](/reference/mgmt/get-version); do **not** gate success on `code == 0`, and do **not** retry-storm during the update.
:::

## When to Use

Use `set_os` to **update the reader's firmware** — during commissioning, a fleet migration, or a security/feature rollout.

- **Host the image** — place the firmware `.DAT` at a URL the reader can reach.
- **Choose authentication** — `NONE` for an open/internal server, or `CERTIFICATE` (with a CA reference or inline PEM) plus a `verificationType` for a secured HTTPS download.
- **Check preconditions** — ensure adequate battery and flash first (low battery → code `14`, insufficient flash → code `8`).
- **Verify afterward** — always confirm the new build with [`get_version`](/reference/mgmt/get-version).

## Key Fields

- **`OSUpdateDetails.url`** — the firmware `.DAT` download URL (a bad/unreachable URL → code `9`).
- **`OSUpdateDetails.authenticationType`** — `NONE` or `CERTIFICATE`; for `CERTIFICATE`, supply `caCertificateFile` (installed reference) or `caCertificateFileContent` (inline PEM).
- **`OSUpdateDetails.verificationType`** — TLS verification of the download server; omit at your peril over HTTPS (see Notes).
- **`response.code`** — `1` means *accepted*, not *updated*; confirm with [`get_version`](/reference/mgmt/get-version). See [Response Codes](#response-codes).

## Firmware update lifecycle

`set_os` is an accept-then-verify cycle, not a single request/response:

1. **Accept** — the device replies `response.code 1` ("Command payload is accepted") on the response topic. This confirms receipt only, and is **identical for success and failure**.
2. **Start event** — a `FIRMWARE_UPDATE` alert appears on the event topic as the download/flash begins:

<JsonBlock title="Event — FIRMWARE_UPDATE (start)" data={{ type: 'NOTIFICATION', timestamp: '2026-06-18T11:30:50.387Z', state: 'ONESHOT', id: 'FIRMWARE_UPDATE', priority: 'HIGH', alertDetails: { fwUpdateStatus: { updateStatus: 'Starting FW update' } } }} />

3. **Device-wide lock** — while the update runs, **every other command** is rejected on the response topic with **`code 4`** ("Firmware update in progress"), echoing the blocked command's name:

<JsonBlock title="Response — another command blocked during the update (code 4)" data={LOCK_JSON} />

4. **Reboot** — the reader reflashes and reboots, and is briefly off the network.
5. **Verify** — run [`get_version`](/reference/mgmt/get-version) and compare `readerVersion.firmwareVersion` against the target build (see the version note below). A failed update leaves the prior version in place; terminal failure surfaces via the `FIRMWARE_UPDATE` event and code `13`, not the `set_os` accept.

## Response payload

The immediate reply is an **accept** — `command`, `requestId`, `apiVersion`, and `response` with **`code 1`**. It confirms receipt, not a completed update.

## Response Codes

`set_os` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `1` | Command payload is accepted (`IOT_STATUS_CMD_PAYLOAD_ACCEPTED`) — **the accept code for this command.** The update proceeds asynchronously; verify with `get_version`. |
| `0` | Success — terminal success; in practice `set_os` acks with `1`, and completion is confirmed via the `get_version` read-back. |
| `4` | Firmware update in progress (`IOT_STATUS_FW_UPDATE_IN_PROGRESS`) — returned to **other** commands while an update is running (the device-wide lock). Pause orchestration until the device returns. |
| `8` | Insufficient flash size (`IOT_ERROR_LOW_FLASH_SIZE`) — not enough free flash to download the firmware. Free space or use a smaller package. |
| `9` | File not found (`IOT_STATUS_FILE_NOT_FOUND`) — the firmware `url` is wrong or unreachable. |
| `13` | Firmware update Failed (`IOT_ERROR_FW_UPDATE_FAIL`) — the download or install failed. Surfaced via the `FIRMWARE_UPDATE` event, not the `set_os` accept. |
| `14` | Battery is low, Cannot update firmware (`IOT_ERROR_FW_UPDATE_FAIL_LOW_BATTERY`) — charge or connect external power, then retry. |
| `101` | Default error — generic catch-all. |

:::note
An invalid `authenticationType` / `verificationType` enum value is rejected with code `23` ("Invalid enum value"). Gate success strictly on the `get_version` read-back, not on the `set_os` response code.
:::

## Notes & Caveats

:::danger
**`code 1` is an accept, not a completed update.** The `set_os` reply is `code 1` for **both** success and failure, and there is no terminal `0`/`13` on the response topic. Confirm the outcome by running [`get_version`](/reference/mgmt/get-version) and comparing the firmware version. A consumer that treats `code == 0` as "updated" will misread every `set_os` reply.
:::

:::warning
**Device-wide lock during the update.** While `set_os` runs, **every** other command returns `code 4` ("Firmware update in progress"), echoing the blocked command's name. Do not retry-storm — pause orchestration until the device re-attaches and `get_version` succeeds. Expect the reader to be unreachable across the reboot.
:::

:::note
**Preconditions.** Ensure the device is ready before issuing: low battery → code `14`; insufficient flash → code `8`; a bad or unreachable `url` → code `9`. `OSUpdateDetails` is required in practice (a request without it validates against the command schema but cannot update).
:::

:::note
**Authentication & transport security.** For `authenticationType: CERTIFICATE`, supply the CA either as an installed reference (`caCertificateFile`, pre-installed via [`install_certificate`](/reference/mgmt/install-certificate)) or inline (`caCertificateFileContent`). `authenticationType: NONE` over an `https://` URL is accepted but **does not verify the server certificate** — for production, use `CERTIFICATE` and/or a `verificationType` (`VERIFY_PEER` / `VERIFY_HOST` / `VERIFY_HOST_PEER`).
:::

:::note
**Confirm by full version string.** `readerVersion.firmwareVersion` is a free-form string. Compare the **whole** value against the target build — the build-variant prefix can change (e.g. `PAAFKS00-013-R02` → `SAAFKS00-013-R02`), so match the full string rather than a substring. See [`get_version`](/reference/mgmt/get-version).
:::

:::note
**`apiVersion` is per-handler.** `set_os` reports `apiVersion` `V1.1` (its response enum is `V1.0` / `V1.1` / `V1.21`), while on the same firmware [`get_version`](/reference/mgmt/get-version) reports `V1.21`. Do not treat it as a single device-wide constant.
:::

:::tip
Sequence a firmware rollout as: pre-check with [`get_status`](/reference/mgmt/get-status) (power/battery) → `set_os` → wait out the reboot (expect `code 4` on other commands) → confirm with [`get_version`](/reference/mgmt/get-version). For the concept-level view, see [System operations](/infrastructure/system-operations).
:::
