## Overview

`delete_wifi_profile` **permanently removes a single saved Wi-Fi profile** from the reader's profile store. The profile is identified by its `essid` inside `wifiProfileInfo`; on success the matching saved profile — including any stored credentials — is deleted, and the command returns a status-only envelope (no profile echo). It is a state-changing command on the device-management (MGMT) endpoint. Confirm the result with [`get_wifi`](/reference/mgmt/get-wifi).

:::warning
`delete_wifi_profile` is **state-changing and irreversible** — there is no "undelete." The saved profile and its stored credentials are gone; to restore it you must re-provision with [`set_wifi`](/reference/mgmt/set-wifi) and re-enter the credentials. However, the deletion itself is **safe for the control session**: the reader **refuses to delete the active (currently-connected) profile** and returns code `16` instead of dropping its own network (see the notes below).
:::

## When to Use

Use `delete_wifi_profile` to **clean up the reader's saved Wi-Fi profiles** — one profile per call, matched by `essid`.

- **Remove an obsolete or incorrect profile** — e.g. a network that no longer exists, or one saved with the wrong credentials.
- **Make room before adding a profile** — the device enforces a maximum number of saved profiles; if [`set_wifi`](/reference/mgmt/set-wifi) `CREATE` returns code `19` (SSID count overflow), delete an unused profile first.
- **Tidy fallback sets** — prune stale entries so the reader only carries the profiles you intend it to roam across.

Delete only a **non-active** profile. To retire the network the reader is currently connected to, first connect it elsewhere (with `set_wifi` `connect: true`) — the device will not delete the profile carrying its live connection (code `16`).

## Key Fields

- **`wifiProfileInfo.essid`** — the network name of the saved profile to delete. Case-sensitive; must exactly match a saved profile (verify the exact name with [`get_wifi`](/reference/mgmt/get-wifi)). Not the active profile, or the delete fails with code `16`.
- **`response.code`** — read this to determine success vs failure: `0` (deleted), `15` (no such saved profile), `16` (that profile is the active connection), `17` (no `essid` supplied). See [Response Codes](#response-codes).

## Response Codes

`delete_wifi_profile` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success — the matching profile was removed. |
| `2` | Invalid payload (`IOT_ERROR_INVALID_PAYLOAD`) — the JSON is malformed or has invalid field values. Validate against the schema and correct. |
| `15` | WIFI Error - SSID not found (`IOT_ERROR_SSID_NOT_FOUND`) — the `essid` is not a saved profile. Verify the exact name (case-sensitive) with `get_wifi`. |
| `16` | WIFI Error - Cannot delete active SSID (`IOT_ERROR_DELETE_ACTIVE_SSID`) — the target `essid` is the currently-connected profile; the reader protects it. Connect elsewhere first, then delete. |
| `17` | WIFI Error - SSID missed (`IOT_ERROR_SSID_MISSED`) — no `essid` was supplied in `wifiProfileInfo`. Include the `essid` field. |
| `101` | Default error — generic catch-all. |

## Notes & Caveats

:::tip
**The active profile is firmware-protected.** Unlike [`set_wifi`](/reference/mgmt/set-wifi) with `connect: true` — which can switch networks and sever the control session — `delete_wifi_profile` **cannot** strand the reader: if you target the currently-connected `essid`, the device refuses with code `16` ("Cannot delete active SSID") rather than dropping its own connection. To retire the network the reader manages itself over, first move it to another profile (`set_wifi` `connect: true`), then delete the now-inactive profile.
:::

:::warning
**Deletion is permanent.** There is no undo. The saved profile and its stored credentials are removed; restoring it means re-provisioning with [`set_wifi`](/reference/mgmt/set-wifi) and re-entering the credentials. Double-check the `essid` before sending — the match is **case-sensitive** and exact.
:::

:::note
**One profile per call, matched by `essid`.** `delete_wifi_profile` removes a single saved profile identified by its network name. A `code 15` means the `essid` is not saved (typo or wrong case); confirm the exact stored name with [`get_wifi`](/reference/mgmt/get-wifi) first. A `code 17` means the request reached the device with no `essid` at all.
:::

:::note
**`wifiProfileInfo` is required in practice, not in the schema.** The command-level schema lists only `command` and `requestId` as `required`, so a request omitting `wifiProfileInfo` would technically pass schema validation — but it names no profile and deletes nothing. Always send `wifiProfileInfo` with a valid `essid` (which **is** required within `wifiProfileInfo`).
:::

:::note
**`apiVersion` is device-emitted and per-handler.** The device returns `apiVersion` `V1.1` on this command, but the response schema does not declare the property (it is accepted, unvalidated). Its practical enum is `V1.0` / `V1.1`. On the same firmware other commands (e.g. [`get_version`](/reference/mgmt/get-version)) report `V1.21`, so do not treat it as a single device-wide constant.
:::

:::tip
Because the response is status-only, **verify the result with [`get_wifi`](/reference/mgmt/get-wifi)** — confirm the deleted `essid` no longer appears in the saved profiles and the connected profile is unchanged. For the concept-level view, see [Wi-Fi networking](/infrastructure/network/wifi).
:::
