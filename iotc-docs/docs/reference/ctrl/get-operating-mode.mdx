## Overview

`get_operating_mode` **reads back the reader's current RFID operating-mode configuration** — the active profile, radio start/stop conditions, Gen2 `query` parameters, select pre-filters, armed access operations, and tag-metadata reporting flags — under `operatingMode.operatingModes`. It is read-only and mutates nothing. It is a **control-plane (CTRL) command** — configure the mode with [`set_operating_mode`](/reference/ctrl/set-operating-mode).

:::warning
This is a **CTRL-endpoint command**: it is published on the **control** endpoint (`zebra/CTRL/clients/cmnd/{serial}`) and is **not answered over the MGMT/MDM channel**. The reader must have an **active CTRL endpoint** (configured with [`config_endpoint`](/reference/mgmt/config-endpoint), `epType: CTRL`, `activate: true`) — otherwise the command gets **no response**. See Topic Mapping.
:::

## When to Use

Use `get_operating_mode` to **inspect or audit the reader's RFID configuration**:

- **Before an inventory** — confirm the profile, `query` parameters, and trigger conditions are what you expect.
- **After `set_operating_mode`** — verify the mode was applied (there is no configuration echo on the set command's response).
- **After a reboot** — the operating mode reverts to default on reboot (see the volatility note); read it back to confirm the current state.
- **Check reporting** — see which `tagMetaDataToEnable` fields the reader will include on tag reads.

| Direction | Topic | Note |
| --- | --- | --- |

| Response (subscribe by app; published by device) | `zebra/CTRL/clients/resp/{serial}` | Correlated reply; the in-payload `requestId` links the twin topics. |

The full pattern is `<tenantId>/<topic>/<deviceSerialNumber>`, where `<topic>` is configured per endpoint. Unlike the MGMT commands, control commands are answered **only on the CTRL endpoint** — publishing `get_operating_mode` to the MGMT/MDM command topic produces no reply. See [Topic hierarchy](/foundations/mqtt/topic-hierarchy) and the [Topic quick reference](/reference/appendices/topic-quick-reference).

:::note
If no CTRL endpoint is active, add or activate one with [`config_endpoint`](/reference/mgmt/config-endpoint) (`epType: CTRL`, `activate: true`) before sending control commands. There is no per-message authentication block; the `requestId` you send is echoed in the response.
:::

## Key Fields

- **`operatingMode.operatingModes.profiles`** — the active profile (e.g. `BALANCED_PERFORMANCE`); `ADVANCED` surfaces `advancedConfigurations`.
- **`query`** — the Gen2 session / inventory-state / SL-flag / tag-population parameters driving inventory.
- **`radioStartConditions` / `radioStopConditions`** — the trigger semantics for starting/stopping inventory.
- **`tagMetaDataToEnable`** — which metadata fields (RSSI, EPC, TID, …) the reader reports on each tag read.
- **`response.code`** — read this to determine success vs failure; note some firmware omits it (see Notes). See [Response Codes](#response-codes).

## Response payload

On success the device returns the envelope (`command`, `requestId`, `apiVersion`, `response`) with the current mode under `operatingMode.operatingModes`. Sub-objects that aren't configured (`advancedConfigurations`, `accessOperations`, `select`) are omitted. The **Example** tab shows a default `BALANCED_PERFORMANCE` read-back.

## Response Codes

`get_operating_mode` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success — the operating mode was retrieved. |
| `3` | Not able to retrieve information (`IOT_ERROR_INFO_NOT_AVAILABLE`) — the device could not gather the operating mode at this time. Retry after a short delay; if persistent, reboot. |
| `101` | Default error — generic catch-all. |

:::note
**Some firmware omits `response.code` on success.** A build was observed returning `response: { "description": "Success" }` with **no `code`** on this command (a known firmware issue, fix in progress; the schema requires `code`). Handle a missing `code` gracefully — treat `description: "Success"` as success, or key on the presence of `operatingMode`.
:::

## Examples

Publish `get_operating_mode` over the CTRL endpoint (with an active CTRL endpoint); the device replies with the snapshot shown under [Response payload](#response-payload). There are no request variants — the command takes no parameters.

## Notes & Caveats

:::warning
**Control-plane routing.** `get_operating_mode` is answered **only on the CTRL endpoint** and requires an **active CTRL endpoint**. It is *not* handled on the MGMT/MDM channel — publishing it there yields no reply. If control commands time out, confirm a CTRL endpoint is active (`get_endpoint_config` → `activeEndpoints`), and activate one with [`config_endpoint`](/reference/mgmt/config-endpoint) if needed.
:::

:::note
**Operating mode is volatile.** The operating mode resets to default (`BALANCED_PERFORMANCE`, armed `accessOperations` cleared) on [`reboot`](/reference/mgmt/reboot). Read it back after a reset to confirm the current state; re-apply with [`set_operating_mode`](/reference/ctrl/set-operating-mode) if you were relying on a non-default configuration.
:::

:::note
**`FAST_READ` is enumerated but not supported.** The `profiles` enum lists `FAST_READ`, but it is **currently not supported** by the firmware (per the schema's own description). Use `CYCLE_COUNT`, `DENSE_READERS`, `OPTIMAL_BATTERY`, `BALANCED_PERFORMANCE`, or `ADVANCED`.
:::

:::note
**Conditional sub-objects.** `advancedConfigurations` appears only when `profiles` is `ADVANCED`; `accessOperations` and `select` appear only when configured. A default read-back typically returns just `profiles`, `radioStartConditions`, `radioStopConditions`, `query`, and `tagMetaDataToEnable`.
:::

:::warning
**`accessOperations` can carry a password.** If access operations are armed, `accessOperations[].config.password` (an 8-hex-char access/lock/kill password) may appear in the read-back — treat it as **sensitive** and mask it in logs and shared artifacts. A default read-back has no `accessOperations` and exposes no credentials.
:::

:::note
**`apiVersion` is per-handler.** `get_operating_mode` reports `apiVersion` `V1.1` (its response enum is `V1.0` / `V1.1`), while on the same firmware [`get_version`](/reference/mgmt/get-version) reports `V1.21`. Do not treat it as a single device-wide constant.
:::

:::tip
Read `get_operating_mode` to confirm the reader's RF configuration before running an inventory with [`control_operation`](/reference/ctrl/control-operation), and to verify a change applied by [`set_operating_mode`](/reference/ctrl/set-operating-mode). For the concept-level view, see [Operating mode](/reference/ctrl/set-operating-mode).
:::
