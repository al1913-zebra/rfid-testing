## Overview

`get_post_filter` **reads back the reader's current post-filters** ‚Äî the rules that include or exclude scanned RFID **tag data** before it is reported to a data endpoint. Each rule matches tag data by pattern (prefix, suffix, or regex) and either reports (`INCLUDE`) or drops (`EXCLUDE`) it, scoped to a data endpoint. It is read-only and mutates nothing. It is a **control-plane (CTRL) command**; configure filters with [`set_post_filter`](/reference/ctrl/set-post-filter).

:::warning
This is a **CTRL-endpoint command**: it is published on the **control** endpoint (`zebra/CTRL/clients/cmnd/{serial}`) and is **not answered over the MGMT/MDM channel**. The reader must have an **active CTRL endpoint** ([`config_endpoint`](/reference/mgmt/config-endpoint), `epType: CTRL`, `activate: true`) or you get no response.
:::

## When to Use

Use `get_post_filter` to **inspect the reader's post-filters**:

- **Audit filtering** ‚Äî see which patterns are included/excluded per data endpoint before relying on the tag-data stream.
- **After `set_post_filter`** ‚Äî verify a filter was applied (the setter's response carries no configuration echo).
- **Confirm none are set** ‚Äî an absent `postFilterPayload` means no filters are configured (see Notes).

:::note
Post-filters act on **tag data** (post-singulation), distinct from the Gen2 `select` pre-filters in [`set_operating_mode`](/reference/ctrl/set-operating-mode) that act on the air interface. If control commands time out, confirm a CTRL endpoint is active and activate one with [`config_endpoint`](/reference/mgmt/config-endpoint) if needed.
:::

## Key Fields

- **`postFilterPayload.postFilter[]`** ‚Äî the configured rules; absent when none are set.
- **`dataEpType`** ‚Äî the data endpoint (`DATA_EP1` / `DATA_EP2`) each rule scopes to.
- **`matchPattern` + `matchPatternMethod`** ‚Äî what to match and how (`PREFIX` / `SUFFIX` / `REGEX`).
- **`reportOperation`** ‚Äî `INCLUDE` (report matches) or `EXCLUDE` (drop matches).
- **`response.code`** ‚Äî read this to determine success vs failure; note the success-path omission (see Notes). See [Response Codes](#response-codes).

## Response Codes

`get_post_filter` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success ‚Äî the post-filter configuration was retrieved (an empty configuration still succeeds). |
| `3` | Not able to retrieve information (`IOT_ERROR_INFO_NOT_AVAILABLE`) ‚Äî the device could not gather the post-filters at this time. Retry after a short delay; if persistent, reboot. |
| `101` | Default error ‚Äî generic catch-all. |

:::note
**On success the device often omits `response.code`**, returning just `response: { "description": "Success" }` (the same behavior as [`get_operating_mode`](/reference/ctrl/get-operating-mode) / [`set_operating_mode`](/reference/ctrl/set-operating-mode)). Treat `description: "Success"` (or an absent `code`) as success ‚Äî don't gate on `code == 0`. The schema still requires `code`.
:::

## Notes & Caveats

:::warning
**Control-plane routing.** `get_post_filter` is answered **only on the CTRL endpoint** and requires an **active CTRL endpoint** ‚Äî it is *not* handled on the MGMT/MDM channel. If it times out, confirm a CTRL endpoint is active (`get_endpoint_config` ‚Üí `activeEndpoints`) and activate one with [`config_endpoint`](/reference/mgmt/config-endpoint).
:::

:::note
**`postFilterPayload` is absent when no filters are set.** With no post-filters configured, the device omits `postFilterPayload` entirely ‚Äî not even an empty `postFilter: []`. Treat its absence as "no filters." When filters exist, `postFilter[]` lists them.
:::

:::note
**Success omits `code`.** On success the device typically returns `response: { "description": "Success" }` with **no `code`** ‚Äî treat an absent `code` / `description: "Success"` as success. Failure would carry a `code` (`3` retrieval failure, `101` generic).
:::

:::note
**Post-filters vs pre-filters.** `get_post_filter` reports **post-filters** ‚Äî pattern rules applied to scanned **tag data** before it reaches a data endpoint (`INCLUDE`/`EXCLUDE`). These are separate from the Gen2 `select` **pre-filters** in [`set_operating_mode`](/reference/ctrl/set-operating-mode), which filter on the air interface during inventory. Configure post-filters with [`set_post_filter`](/reference/ctrl/set-post-filter).
:::

:::note
**`apiVersion` is per-handler.** `get_post_filter` reports `apiVersion` `V1.1` (its response enum is `V1.0` / `V1.1`), while on the same firmware [`get_version`](/reference/mgmt/get-version) reports `V1.21`. Do not treat it as a single device-wide constant.
:::

:::tip
`get_post_filter` is a read-only, idempotent getter ‚Äî safe to poll. Use it to verify a change applied by [`set_post_filter`](/reference/ctrl/set-post-filter); it carries no credentials in request or response. For the concept-level view, see [Tag filtering](/reference/ctrl/set-post-filter) and [Post-filters](/rfid/post-filters).
:::
