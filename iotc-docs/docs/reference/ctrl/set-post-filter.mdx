> 📕 **REFERENCE** · **Audience:** API consumer · **Use:** add, delete, or modify a post-filter rule on the reader

## Overview

`set_post_filter` **configures a post-filter** — a rule that includes or excludes scanned RFID **tag data** before it is reported to a data endpoint. Each call operates on **one** filter, chosen by `postFilterPayload.operation` (`ADD` / `DELETE` / `MODIFY`): the rule matches tag data by pattern (prefix, suffix, or regex) and either reports (`INCLUDE`) or drops (`EXCLUDE`) it, scoped to a data endpoint. It is a state-changing **control-plane (CTRL) command**; read the current filters back with [`get_post_filter`](/reference/ctrl/get-post-filter).

:::warning
This is a **CTRL-endpoint command**, published on the **control** endpoint (`zebra/CTRL/clients/cmnd/{serial}`) and **not answered over the MGMT/MDM channel**. The reader must have an **active CTRL endpoint** ([`config_endpoint`](/reference/mgmt/config-endpoint), `epType: CTRL`, `activate: true`) or you get no response. Also: `dataEpType` is the **enum** `DATA_EP1`/`DATA_EP2`, **not** the endpoint *name* — sending an endpoint name returns code `23` (see Notes).
:::

## When to Use

Use `set_post_filter` to **control which scanned tag data reaches a data endpoint**:

- **Include only matching tags** — `reportOperation: INCLUDE` reports only tags whose data matches; everything else is dropped.
- **Exclude noisy tags** — `reportOperation: EXCLUDE` drops matching tags from reporting.
- **Match by prefix, suffix, or regex** — choose `matchPatternMethod` to fit the tag-data pattern.
- **Manage the rule set** — `ADD` to install, `DELETE` to remove, `MODIFY` to update; verify with [`get_post_filter`](/reference/ctrl/get-post-filter).

## Key Fields

- **`postFilterPayload.operation`** — `ADD` / `DELETE` / `MODIFY`; determines the action on the rule set.
- **`dataEpType`** — the data-endpoint enum (`DATA_EP1` / `DATA_EP2`) the filter binds to — ⚠️ not the endpoint name (see Notes).
- **`matchPattern` + `matchPatternMethod`** — what to match and how (`PREFIX` / `SUFFIX` / `REGEX`).
- **`reportOperation`** — `INCLUDE` (report matches) or `EXCLUDE` (drop matches).
- **`response.code`** — read this to determine success vs failure; note the success-path omission (see Notes). See [Response Codes](#response-codes).

## Operations

`postFilterPayload.operation` selects the action; each call handles one rule:

- **`ADD`** — install a new post-filter for the given `dataEpType`.
- **`DELETE`** — remove an existing filter (match its parameters). Undoes a prior `ADD`.
- **`MODIFY`** — update an existing filter.

Confirm the resulting rule set with [`get_post_filter`](/reference/ctrl/get-post-filter), which returns the stored `postFilter[]` (or nothing when none remain).

## Response payload

`set_post_filter` returns a **status-only** envelope — `command`, `requestId`, `apiVersion`, and `response`. There is no filter echo; confirm with [`get_post_filter`](/reference/ctrl/get-post-filter).

`set_post_filter` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success — the filter was added, deleted, or modified. (Often conveyed as `description: "Success"` with `code` omitted — see above.) |
| `23` | Invalid enum value (`IOT_ERROR_INVALID_ENUM`) — a field (`operation`, `dataEpType`, `matchPatternMethod`, or `reportOperation`) is outside its allowed set. **Commonly triggered by sending an endpoint _name_ (e.g. `DATA1_EP`) as `dataEpType` instead of the enum `DATA_EP1` — see Notes.** |
| `101` | Default error — generic catch-all. |

## Notes & Caveats

:::warning
**`dataEpType` is an enum, not the endpoint name.** `dataEpType` must be the enum value `DATA_EP1` or `DATA_EP2` — **not** the endpoint *name* you gave a data endpoint in [`config_endpoint`](/reference/mgmt/config-endpoint) (e.g. `DATA1_EP`). Sending the endpoint name is an enum violation and the device rejects it with **code `23`** ("Invalid enum value"). Map your `DATA1`/`DATA2` endpoint to `DATA_EP1`/`DATA_EP2` when setting a filter.
:::

:::note
**`matchPattern` rules (description-only, not schema-enforced).** For `PREFIX` and `SUFFIX` methods, `matchPattern` must be **hexadecimal digits only, with an even count**; for `REGEX` it is a free-form regular expression. Also, **a `PREFIX` post-filter precludes Gen2 `select` pre-filters** — you can't use both at once. These rules aren't machine-validated, so malformed patterns may be accepted and simply not match.
:::

:::note
**Success omits `code`; errors carry one.** On success the device typically returns `response: { "description": "Success" }` with **no `code`** (the same behavior as the other control commands). An error returns a `code` — e.g. `23` for an invalid enum value. Treat an absent `code` / `description: "Success"` as success; verify the applied rule with [`get_post_filter`](/reference/ctrl/get-post-filter).
:::

:::note
**Post-filters vs pre-filters.** `set_post_filter` configures **post-filters** — rules applied to scanned **tag data** before it reaches a data endpoint (`INCLUDE`/`EXCLUDE`). These are separate from the Gen2 `select` **pre-filters** in [`set_operating_mode`](/reference/ctrl/set-operating-mode), which filter on the air interface during inventory.
:::

:::note
**`apiVersion` is per-handler.** `set_post_filter` reports `apiVersion` `V1.1` (its response enum is `V1.0` / `V1.1`), while on the same firmware [`get_version`](/reference/mgmt/get-version) reports `V1.21`. Do not treat it as a single device-wide constant.
:::

:::tip
`set_post_filter` is **reversible** — an `ADD` is undone by a matching `DELETE`. It carries no credentials (match patterns are tag-data hex/regex strings). After setting a filter, **verify with [`get_post_filter`](/reference/ctrl/get-post-filter)** — the setter's response has no configuration echo. For the concept-level view, see [Tag filtering](/reference/ctrl/get-post-filter) and [Post-filters](/rfid/post-filters).
:::
