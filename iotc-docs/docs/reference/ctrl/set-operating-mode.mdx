## Overview

`set_operating_mode` **configures the reader's RFID operating mode** — the profile, advanced RF configuration, tag access operations, radio start/stop triggers, the Gen2 `query`, select pre-filters, and tag-metadata reporting flags. It is a state-changing **control-plane (CTRL) command**. It is a **partial, merging** update: you send only the sub-objects you want to change and they merge into the current mode. Read the result back with [`get_operating_mode`](/reference/ctrl/get-operating-mode).

:::danger
**`accessOperations` arm destructive tag operations.** `WRITE` rewrites tag memory, `LOCK` with `PERMANENT_LOCK`/`PERMANENT_UNLOCK` is **irreversible**, and `KILL` **permanently disables the tag**. These operations are *armed* by `set_operating_mode` and **execute on the next inventory** — against every tag in range. Clear armed operations before running an inventory you don't intend them for: **overwrite the `accessOperations` array with a benign (or empty) set, or [`reboot`](/reference/mgmt/reboot)**. A profile-only (or any partial set that omits `accessOperations`) does **not** clear them — the merge preserves them (see [Partial updates and merge](#partial-updates-and-merge)). Also note: **on success this command often omits `response.code`** — do not gate success on `code == 0` (see below).
:::

:::warning
This is a **CTRL-endpoint command**, published on the **control** endpoint (`zebra/CTRL/clients/cmnd/{serial}`) and **not answered over the MGMT/MDM channel**. The reader must have an **active CTRL endpoint** ([`config_endpoint`](/reference/mgmt/config-endpoint), `epType: CTRL`, `activate: true`) or you get no response.
:::

## When to Use

Use `set_operating_mode` to **configure how the reader reads (and acts on) tags**:

- **Select a profile** — e.g. `CYCLE_COUNT`, `DENSE_READERS`, `OPTIMAL_BATTERY`, `BALANCED_PERFORMANCE`, or `ADVANCED` (with `advancedConfigurations`).
- **Tune the radio** — transmit power, link profile, session, dynamic power (ADVANCED).
- **Compose triggers** — `radioStartConditions` / `radioStopConditions` (trigger, delay, tag/time/cycle stops).
- **Shape inventory** — Gen2 `query` and up to 32 `select` pre-filters.
- **Choose reporting** — `tagMetaDataToEnable` flags.
- **Arm tag operations** — `accessOperations` (READ/WRITE/ACCESS/LOCK/KILL) that run on the next inventory. Handle with care (see the danger note).

## Key Fields

- **`operatingModes.profiles`** — the RF profile; `ADVANCED` requires `advancedConfigurations`.
- **`operatingModes.accessOperations`** — ⚠️ armed tag operations (destructive; see the danger note).
- **`operatingModes.select`** — up to 32 Gen2 pre-filters.
- **`operatingModes.query` / `radioStartConditions` / `radioStopConditions`** — inventory shaping and trigger semantics.
- **`response.code`** — read this to determine success vs failure; note the success-path omission and the partial-apply-on-error behavior (see Notes). See [Response Codes](#response-codes).

## Partial updates and merge

`set_operating_mode` is **cumulative**: it updates only the aspects present in the payload and leaves all others intact — it is **not** a full replacement. For example, sending only `radioStartConditions` changes the triggers while preserving `profiles`, `query`, `select`, `accessOperations`, and `tagMetaDataToEnable`. Send `{profiles: "BALANCED_PERFORMANCE"}` alone to reset the profile without touching anything else. Because merges accumulate, **armed `accessOperations` persist** across later partial sets until you overwrite them or reboot. Confirm the resulting merged mode with [`get_operating_mode`](/reference/ctrl/get-operating-mode).

:::note
**On success the device often omits `response.code`**, returning just `response: { "description": "Success" }`. Conversely, an error carries a `code` (e.g. `101`). ## Response Codes

`set_operating_mode` returns `response.code` per `error_codes.json`. The codes applicable to this command are below; for the full table see the [Error codes reference](/reference/errors/codes) and [Error format](/reference/errors/format).

| Code | Description |
| --- | --- |
| `0` | Success — the mode was applied. (Often conveyed as `description: "Success"` with `code` omitted — see above.) |
| `11` | Inventory in progress (`IOT_ERROR_INVENTORY_IN_PROGRESS`) — the mode can't be changed while an inventory is running. Stop it with [`control_operation`](/reference/ctrl/control-operation) first. |
| `22` | Advanced configuration not set (`IOT_CTRL_ADVANCED_PROFILE_NOT_SET`) — `profiles: ADVANCED` was sent without `advancedConfigurations`. |
| `23` | Invalid enum value (`IOT_ERROR_INVALID_ENUM`) — a field (e.g. `profiles`, `operationType`, a Gen2 enum) is outside its allowed set. |
| `24` | Max 32 prefilters limit exceeded (`IOT_ERROR_PREFILTERS_LIMIT_EXCEEDED`) — more than 32 `select` entries. |
| `28` | Tag match pattern length exceeded (`IOT_ERROR_TAG_MATCH_PATTERN_LENGTH_EXCEEDED`) — a `select.tagPattern` is too long. |
| `31` | Max write length exceeded (`IOT_ERROR_MAX_WRITE_LENGTH_EXCEEDED`) — a WRITE `data` block is too large. |
| `32` | Write data length mismatch (`IOT_ERROR_WRITE_DATA_LENGTH_MISMATCH`) — a WRITE `data` length doesn't match its `length`. |
| `101` | Default error (`Error in processing command`) — a generic failure. **Note:** a `101` can still leave the mode *partially applied* — read back with `get_operating_mode` to see what took effect. |

## Notes & Caveats

:::danger
**Access operations are destructive and armed for the next inventory.** `accessOperations` (`WRITE`, `LOCK`, `KILL`) are stored by `set_operating_mode` and executed against tags **on the next inventory START** — not immediately. `KILL` permanently disables a tag; `LOCK` with `PERMANENT_LOCK`/`PERMANENT_UNLOCK` is irreversible; `WRITE` overwrites tag memory. Because partial sets **merge**, armed operations persist until you **overwrite the `accessOperations` array** (with a benign or empty set) or [`reboot`](/reference/mgmt/reboot). A profile-only set — or any partial set that omits `accessOperations` — does **not** clear them. Do this before running an inventory that shouldn't trigger them.
:::

:::warning
**Credentials.** `accessOperations[].config.password` is an 8-hex-char (32-bit) access/lock/kill password you send (default `0000`) — mask it in shared artifacts. The device stores it but never echoes it back ([`get_operating_mode`](/reference/ctrl/get-operating-mode) returns it empty). This command carries no broker/Wi-Fi credentials.
:::

:::note
**Success omits `code`; errors carry one; `101` may partially apply.** On success the device typically returns `response: { "description": "Success" }` with **no `code`** — treat an absent `code` / `description: "Success"` as success. An error returns a `code` (e.g. `101` "Error in processing command"), and a `101` can leave the mode **partially applied**, so always verify the result with [`get_operating_mode`](/reference/ctrl/get-operating-mode) rather than assuming a `101` changed nothing.
:::

:::note
**Partial-merge semantics.** Only the `operatingModes` sub-objects present in the payload change; the rest are preserved. See [Partial updates and merge](#partial-updates-and-merge).
:::

:::note
**`ADVANCED` requires `advancedConfigurations`; limits are enforced.** `profiles: ADVANCED` without `advancedConfigurations` returns code `22`. `select` allows up to 32 filters (code `24` if exceeded); an overlong `select.tagPattern` returns `28`; WRITE `data` that is too large or mismatched returns `31` / `32`. `FAST_READ` is enumerated but **not currently supported**, and configuring `FAST_READ` disables data events.
:::

:::note
**Metadata flags auto-provision READ operations.** Enabling `tagMetaDataToEnable.EPC` / `TID` / `USER` causes the device to auto-add a generic `READ {offset 0, length 0}` access operation for that bank (and disabling it removes that op). So a `get_operating_mode` read-back may show READ operations you didn't explicitly send.
:::

:::note
**Volatile & CTRL-only, per-handler `apiVersion`.** The operating mode resets to default on [`reboot`](/reference/mgmt/reboot). The command is answered only on the CTRL endpoint (see Topic Mapping). `set_operating_mode` reports `apiVersion` `V1.1`, while [`get_version`](/reference/mgmt/get-version) reports `V1.21` — per-handler, not device-wide.
:::

:::tip
Configure with `set_operating_mode`, then **verify with [`get_operating_mode`](/reference/ctrl/get-operating-mode)** (especially after a `101`), and run the inventory with [`control_operation`](/reference/ctrl/control-operation). For the concept-level view, see [Operating-mode profiles](/rfid/operating-mode-profiles).
:::
