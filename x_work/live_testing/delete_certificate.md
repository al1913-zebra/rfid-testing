# Command: delete_certificate

> **Test, validation & divergence report.** Live-exercised against RFD40 Premium+ serial `RFD40-24190525100255`, broker `10.117.229.9:1883` (plain MQTT, tenant `zebra`, endpoint `MDM_EP`). All three operator payloads were run with before/after read-backs, plus error/edge probes. Generated 2026-06-17.

## 1. Intent & Objective

`delete_certificate` is a **state-changing** device-management (MDM-plane) command that removes a certificate from the device trust store, identified by `certificateInfo.{name, type}`. It is **synchronous** — success returns `response.code 0` and the certificate is gone (verified by read-back), a non-existent target returns `code 21`. **`name` is optional**: per the payload schema, omitting it deletes **ALL** certificates of the matching `type` (a bulk, destructive operation). `type` is the only required field. [verified-on-device: delete of an existing cert → code 0 + removal; non-existent → code 21]

## 3. Request Payload Breakdown

Envelope: `command`, `requestId`, `certificateInfo`. The command schema has **no top-level `required`** (so `certificateInfo` is optional in-schema, though the device requires it — see §7). `certificateInfo` (`delCertPayload.yaml`):

| Field | Type | Required | Enum / Constraint | Notes |
|---|---|---|---|---|
| `name` | string | optional | — | The cert to delete. **If omitted → ALL certs of `type` are deleted** (bulk). |
| `type` | string | **Yes** | `server / client / mqtt / wifi / filestore` (no leading space) | Cert category. Enforced live (missing → `code 101`). |

**The three operator payloads (= the schema's own bundled examples):**
```json
{ "command":"delete_certificate", "requestId":"abc123", "certificateInfo":{ "name":"ca_cert",     "type":"wifi" } }      // #1 wifi CA
{ "command":"delete_certificate", "requestId":"abc123", "certificateInfo":{ "name":"client_cert", "type":"mqtt" } }      // #2 mqtt client
{ "command":"delete_certificate", "requestId":"abc123", "certificateInfo":{ "name":"ca_cert",     "type":"filestore" } } // #3 filestore CA
```

## 4. Response Payload Breakdown

Envelope: `command`, `requestId`, `apiVersion` (enum `[V1.0,V1.1]`, live `V1.1`), `response` (`$ref response.yaml`). The response schema **does** declare `required: [command, requestId, apiVersion, response]`.

**Success (LIVE, verbatim):**
```json
{ "command":"delete_certificate", "requestId":"abc123", "apiVersion":"V1.1", "response":{ "code":0, "description":"Success" } }
```

**Not-found (LIVE, verbatim):**
```json
{ "command":"delete_certificate", "requestId":"abc123", "apiVersion":"V1.1", "response":{ "code":21, "description":"File not found" } }
```
> Note `code 21`'s description is **"File not found"**, but the schema defines code 21 = "Certificate not found" ("File not found" is code 9). See §7.

### 7.1 Addendum — trust-store cleanup & the type-null orphan trap (this run)

Per operator request the trust store was cleaned. It held `ca_cert/mqtt` plus two `tNoType_*` strays of `type: null` (left by an earlier `install_certificate` missing-`type` probe — see install_certificate.md NEW-1). Outcome:

| Target | Action | Result |
|---|---|---|
| `ca_cert` / `mqtt` | `delete_certificate {name:ca_cert, type:mqtt}` | **`code 0` Success — removed** |
| `tNoType_ca_cert` / null | `delete_certificate` tried with `type` = mqtt, wifi, filestore, server, client | **all 5 → `code 21` "File not found" — UN-DELETABLE** |
| `tNoType_client_cert` / null | same (all 5 types) | **all 5 → `code 21` — UN-DELETABLE** |

**DC-22 (CRITICAL, Behavioral Divergence) — type-null certs are un-deletable via `delete_certificate`.** Delete matches on `{name, type}` and *requires* a `type` enum value, but a cert installed with `type: null` (which `install_certificate` accepts) matches no `type`, so every delete attempt returns `code 21`. **The documented delete API cannot remove a cert the documented install API can create** — a permanent orphan through the normal API surface. *Fix (either/both):* (a) `install_certificate` must enforce required `type` so typeless certs can't be created; (b) `delete_certificate` should allow deletion by `name` alone (or accept a wildcard/`null` type) so any installed entry is removable.

**Recovery workaround that cleared them.** Re-installing the same name with an explicit type — `install_certificate {name:"tNoType", type:"mqtt", certSource:DIRECT, certificateBundle:{ca_cert,client_cert,client_key}}` — removed both strays: the read-back went **2 → 0**. (Re-installing an existing name appears to replace/clear the prior typeless entries; the install ACK itself was not captured this run, but two consecutive read-backs confirmed the entries were gone.)

**Final state: trust store EMPTY (0 certificates) — CLEAN**, confirmed by two consecutive `get_installed_certificates` read-backs (`certInfo: []`). An independent confirmation read-back was retried after the host laptop changed WiFi (broker still reachable, `rc=0`, host egress IP `10.233.46.168` → `10.233.46.75`), but a 90 s poll got no device response — so the blocker is the **device's own MDM session** (device-side intermittency: off-WiFi / asleep / session churn), not the host network. The clean state stands on the two cleanup-session read-backs; re-confirm whenever the sled's session is back up.

## 8. Safety / Operational Notes

- **Destructive & synchronous.** A successful `delete_certificate` immediately removes the cert (verified: `client_cert/mqtt` was deleted, 4→3). There is no undo — recovery means re-installing via `install_certificate`.
- **`name`-omitted = bulk delete.** Per `delCertPayload.yaml`, omitting `name` deletes **every** certificate of the given `type`. Always send an explicit `name` unless a full type-wipe is intended. (Not exercised live — too destructive.)
- **Type-null certs are orphaned.** A cert installed without a `type` (`type: null`) **cannot be removed** by `delete_certificate` (it matches no `type` enum → `code 21`). Avoid creating them (always send `type` to `install_certificate`); if one exists, the only API recovery observed is to re-install the same `name` with an explicit `type`, which clears it (see §7.1).
- **Don't delete a cert in active use.** On this plain-MQTT (1883, no TLS) deployment the mqtt certs are not in the live trust path, so deleting them was safe; on a TLS endpoint, deleting the active broker/client cert would break the management connection.
- **Error semantics to code against:** existing→`0`; non-existent→`21` (but described "File not found"); invalid `type`→`23`; missing `type`/`certificateInfo`→**`101`** (undocumented, out-of-range). Gate success on `code == 0`.
