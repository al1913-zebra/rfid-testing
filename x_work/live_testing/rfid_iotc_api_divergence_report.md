### 2.3 Analysis & verification
A two-stage pipeline ran per operation: (1) an **extraction** agent read the `.md` plus every schema file it cites and pulled every documented divergence at one-row-per-(property x defect) granularity; (2) an **adversarial verification** agent re-read the same files on disk, re-confirmed each schema fact, preserved every live device value verbatim, dropped any claim not actually documented in the `.md`, and re-scanned the whole file for missed divergences. Notable reconciliations are logged in Appendix A. Selected high-impact schema facts were additionally re-confirmed by hand against the schema files on disk.

### 3.4 Critical findings (6) — read first

- **GAP-H-0001 — control\_operation** (Semantic Schema Error): The START success command response omits response.code, emitting only {description: "Success"}. The inner response object therefore violates response.yaml's required [code, description] — strict validation of the success response always fails.
- **GAP-H-0002 — control\_operation** (Semantic Schema Error): The STOP success command response also omits response.code, emitting only {description: "Success"}. The inner response object violates response.yaml's required [code, description], so the STOP success response also fails strict validation (a second distinct observed message exhibiting the omission).
- **GAP-H-0005 — get\_operating\_mode** (Missing Property): The live device response omits the required response.code field, conveying success only via description: "Success". Because response/control/get\_operating\_mode.json requires the response object and response.yaml requires code, a response object without code fails strict schema validation. This is a firmware defect (the schema correctly requires code); every other command this session returned a numeric response.code.
- **GAP-H-0041 — config\_endpoint** (Semantic Schema Error): Firmware returned response.code 101, which exceeds the response schema's maximum of 30. The live error envelope fails response-schema validation because code 101 is out of the declared 0..30 range and absent from the documented code table.
- **GAP-H-0083 — get\_endpoint\_config** (Property Divergence): The response schema declares the endpoint payload under property `epDetails` ($ref endpointResponse.yaml), but both the schema's own examples and the live device emit the key `endpointResponse`. Because the schema has no `additionalProperties: false` and no top-level `required`, the device key is accepted as an unvalidated extra property and the declared `epDetails` validator never runs, so the entire endpoint configuration escapes validation (masking E3/E4). The bypass scales linearly with active-endpoint count (5 masked errors with 1 active endpoint, 10 with 2).
- **GAP-H-0118 — get\_status** (Semantic Schema Error): The required array lists 'systemTime' (no space) but no property of that name exists in properties (the property key is 'systemTime ' with a trailing space). The required entry is therefore permanently unsatisfiable: a strict validator rejects EVERY response regardless of what the device sends.

> **Cross-cutting pattern.** The most pervasive defect family is the **success-path omission of `response.code`**: multiple commands return `{"description":"Success"}` with `code` absent, which is INVALID against `refrence/response/response.yaml` `required: [code, description]`. A single firmware fix (always emit `response.code`, `0` on success) clears the bulk of the CRITICAL/MAJOR `Semantic Schema Error` rows.
