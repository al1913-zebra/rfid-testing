# Command: install_certificate

> **Test, validation & divergence report.** Live-exercised against RFD40 Premium+ serial `RFD40-24190525100255`, broker `10.117.229.9:1883` (plain MQTT, tenant `zebra`, endpoint `MDM_EP`). Generated 2026-06-16; updated 2026-06-17 with an operator-supplied HTTP run that surfaced `FILE_DOWNLOAD` events on the `…/event` topic (see §2, §7.4, and divergences ICN-1…ICN-5).

## 1. Intent & Objective

`install_certificate` is a **device-management (dev_mgmt / MDM-plane)** command that installs a certificate or key onto the reader's trust store — used for MQTT-TLS brokers, Wi-Fi EAP, file-store, and client/server auth. It supports two delivery modes: **`DIRECT`** (certificate content embedded inline in `certificateBundle`) and **`HTTP`** (the device downloads the content from URLs in `url[]`). It is an **asynchronous-accept** command: the device validates the envelope and replies `response.code 1` ("Command payload is accepted") and then applies the certificate in the background; completion is confirmed out-of-band via `get_installed_certificates`. [verified-on-device: code 1 ack for both DIRECT and HTTP modes; read-back via get_installed_certificates]

## 4. Response Payload Breakdown

The command returns an **accept acknowledgement**, not a terminal success:

```json
{ "command": "install_certificate", "requestId": "cert-test-003", "apiVersion": "V1.21",
  "response": { "code": 1, "description": "Command payload is accepted" } }
```

- `response.code` = **1** ("Command payload is accepted") on success — **not** `0` ("Success"). [verified-on-device, all attempts/modes]
- `apiVersion` returned = **`V1.21`** (out-of-enum vs schema `[V1.0, V1.1]`). [verified-on-device]
- **Completion is confirmed via `get_installed_certificates`** (the only way to know the install finished):

```json
{ "command": "get_installed_certificates", "requestId": "...", "apiVersion": "V1.1",
  "installedCerts": { "certInfo": [
    { "name": "ca_cert",     "type": "mqtt", "serial": "7d:ef:d4:c3:2d:39:4a:0b", "subjectName": "Zebra Technologies Root Certificatio", "publicKeyAlg": "RSA", "validFrom": "2024-07-01T07:00:00Z", "validTill": "2034-01-19T07:37:08Z" },
    { "name": "client_cert", "type": "mqtt", "serial": "3b:ab:a2:97:0e:dc:30:8f", "subjectName": "client", "publicKeyAlg": "RSA", "validFrom": "2024-07-01T07:00:00Z", "validTill": "2034-01-19T07:37:52Z" }
  ] },
  "response": { "code": 0, "description": "Success" } }
```

## 8. Safety / Operational Notes

- **State-changing.** `install_certificate` writes to the device trust store; reversible via `delete_certificate`. Confirm the installed set afterward with `get_installed_certificates`.
- **MQTT-cert installs touch the management transport.** Installing a `type: mqtt` certificate alters the credentials the device's MQTT/TLS connections use. During testing the device stopped responding on the broker for a period after the install; the operator reports the MDM session itself remained established. Treat post-install responsiveness as something to **monitor / re-verify**, and avoid installing an mqtt cert that would invalidate the active management connection.
- **DIRECT sends the private key in cleartext** over plain MQTT (1883, no TLS). Acceptable for throwaway test certs only; for production keys use an MQTT-TLS (8883) broker or the HTTP-over-HTTPS path.
- **HTTP mode needs device→host reachability** (routing + inbound firewall for the file-server port). When that's not available, **`DIRECT` is the firewall-free alternative** and was used to complete the install here.
