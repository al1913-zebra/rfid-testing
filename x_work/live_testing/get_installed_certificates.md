# Command: get_installed_certificates

> **Test, validation & divergence report.** Live-exercised against RFD40 Premium+ serial `RFD40-24190525100255`, broker `10.117.229.9:1883` (plain MQTT, tenant `zebra`, endpoint `MDM_EP`). This run captured a **populated** certificate list (2 certs) **and a live error path** — superseding the earlier doc whose live list was empty. Generated 2026-06-17.

## 1. Intent & Objective

`get_installed_certificates` is a **read-only** device-management (MDM-plane) command that lists the certificates in the device trust store, returning per-certificate **metadata** (name, type, issuer/subject, serial, public-key algorithm, validity dates) under `installedCerts.certInfo[]`. It mutates nothing and exposes **no private-key material**. The request is a minimal `command`+`requestId` envelope. **Note the canonical literal is plural** — `get_installed_certificates` — even though the schema file/title is the singular `get_installed_certificate` (the singular name is **not** dispatched by the device). [verified-on-device: plural returns code 0 with a 2-entry cert list; singular gets no reply]

## 8. Security / Operational Notes

- **Read-only, metadata only.** Returns public certificate metadata (issuer/subject/serial/validity/public-key alg) — **no private keys**. Still, a cert inventory reveals the device's PKI/trust topology; redact in shared logs/screenshots.
- **Use the plural literal.** Publish `command: "get_installed_certificates"`; the singular `get_installed_certificate` (the schema file/title) is **not** dispatched.
- **Validation will fail on strict schemas.** Every live `certInfo` entry violates `installedCertResponse.yaml` (leading-space `type` enum, `publicKeyAlg` enum, missing `installTime`). A strict validator rejects real device output; relax/fix the schema (see §7) before enforcing it in a gateway.
- **Closed-schema parser.** The device rejects unknown request fields (`code 2`) though the schema is open — build requests to the exact field set.
