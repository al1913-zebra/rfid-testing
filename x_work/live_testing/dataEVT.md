# Event: dataEVT

## 1. Intent & Objective

dataEVT is a Data Event that captures and represents the structured information related to various events generated by the system, which may include tag data, barcode data, and other relevant event-related details for review in real-time or during post-event reviews [verified-from-schema: events/dataEVT.json description]. The schema title is "Data Events" with x-stoplight id "ba1c453601e2c" [verified-from-schema: events/dataEVT.json]. dataEVT is an event the device PUBLISHES; no operator command targets it directly [verified-from-test-harness: deployment topology — CTRL_EP active; DATA1_EP active; CONTROL routing on the cmnd/resp topics]. The stream is triggered by an active RFID inventory started with a control_operation START command on the control plane [verified-on-device: RFD40 serial 24190525100255]. The device read real physical tags and the START -> dataEVT stream -> STOP lifecycle worked end-to-end [verified-on-device: RFD40 serial 24190525100255].

The objective of this document is to describe the on-wire topic, the payload schema, and the live behavior of dataEVT, documenting both a default-operating-mode capture and an enriched-metadata worked example [verified-on-device: RFD40 serial 24190525100255]. Each tagData entry's field set is gated by the device operating mode's tagMetaDataToEnable flags [verified-on-device: RFD40 serial 24190525100255].

## 6. Conformance & Spec Notes

**D1 — Live tagData OMITS the schema-required `format` field.** tagDataEVTs.yaml declares required [eventNum, format] [verified-from-schema: refrence/events/tagDataEVTs.yaml required]. Neither the default nor the enriched live tagData entries ever carried `format`, so live tag-data items are INVALID against their own item-schema required (= control_operation C5) [verified-on-device: RFD40 serial 24190525100255]. Fix: report `format`, or drop it from required [inferred-from-live: a field that is required-but-never-emitted can be reconciled only by emitting it or relaxing the requirement].

**D2 — Top-level required satisfied.** The dataEVT top-level required is only [timestamp] [verified-from-schema: events/dataEVT.json required]. Live events always carried type, timestamp, and data, which satisfies it [verified-on-device: RFD40 serial 24190525100255]. dataEVT.type echoed the active profile "BALANCED_PERFORMANCE" on every event [verified-on-device: RFD40 serial 24190525100255].

**D3 — tagData fields are GATED by tagMetaDataToEnable, proven live in BOTH directions.** Default mode yielded only EPCid/eventNum/peakRssi/seenCount, while enriched mode additionally yielded EPC/TID/PC/channel/phase [verified-on-device: RFD40 serial 24190525100255].
