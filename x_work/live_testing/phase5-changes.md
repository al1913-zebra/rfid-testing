## Considered but NOT changed (logged in BLOCKERS_AND_INCONSISTENCIES.md instead)
- **`SEEN_COUNT` (examples) vs `SEENCOUNT` (property)** in `operatingModePayload.yaml` — explicitly *log, do not fix* (no `additionalProperties:false`, so it validates silently). 
- **`get_installed_certificate.json` (file, singular) vs `get_installed_certificates` (enum/command value, plural)** — a rename would change the generated `operationId`/path; the command *literal* is already consistently plural. Logged, not renamed.
- **Partial command examples** (`config_endpoint`, `set_config`, `set_wifi`) omitting required `configuration`/`wifiConfig` fields; **placeholder values** (`linkProfile: string`, `securityParams.algorithm: ToDo`, `mask: string`); **`session: S0`** vs enum `SESSION_0..3`; **`status: Disabled`/`Connected`** case mismatches — these are example-quality issues; fixing examples blindly is out of scope (logged).
- **Duplicate `heartbeatEvents.yaml` / `heartbeatEvents_new.yaml`**, **orphan responses**, **`requestId` 16-hex-vs-examples**, **named-key vs `payload.oneOf`** — structural/by-design; logged.
- `maxItems: 3/1` was briefly added to publish/subscribeTopics then **reverted** — it is an opinionated constraint absent from the original schema and would flag the existing `set_config` ex#4 (3 subscribe topics). The limits remain documented in the array `description` and enforced by firmware codes 25/26.
